Amazon Redshift の概念実証の実施

Amazon Redshift は、高速でスケーラブルなデータウェアハウスです。シンプルで費用対効果の高さが特長であり、お客様はすべてのデータを標準 SQL を使って既存のビジネスインテリジェンス (BI) ツールで分析できます。Amazon Redshift は、低コストのクラウドデータウェアハウスで高速なパフォーマンスを提供します。高度なクエリの最適化、高速化されたキャッシュ、ハイパフォーマンスローカルディスクの列指向ストレージ、および大量の並行クエリの実行が使用されます。

以下のセクションでは、Amazon Redshift の概念実証を構築するためのフレームワークについて説明します。このフレームワークでは、安全でパフォーマンスが高く、コスト効果が高いデータウェアハウスを設計、運用するためのアーキテクチャのベストプラクティスを使用します。このガイダンスは、さまざまなビジネスの種類とユースケースにおけるお客様の数千のアーキテクチャ設計を確認して作成されています。データウェアハウスのワークロードを評価するための条件を開発できるように、お客様の体験をまとめてこの一連のベストプラクティスを作成しました。

プロセスの概要

概念実証の実施は、3 つのステップのプロセスです。

  1. 概念実証の目標を特定する – ビジネス要件と成功基準から遡って、それらを技術的な概念実証のプロジェクト計画に変換できます。

  2. 概念実証環境をセットアップする – セットアッププロセスのほとんどは、リソースを作成するボタンを数回クリックするだけで済みます。数分以内に、データをロードした状態でデータウェアハウス環境を準備できます。

  3. 目標を確実に達成するために、概念実証プロジェクト計画を完了させます。

以下のセクションでは、各ステップの詳細について説明します。

ビジネス目標と成功基準を特定する

PoC (概念実証) の目標の確認は、評価プロセスの一部として測定する内容を決定するうえで重要な役割を担います。評価条件には、現在のスケーリング課題、データウェアハウスのカスタマーエクスペリエンスを改善するために行うことを希望する機能強化、および現在の業務上の課題への対応方法を含める必要があります。次の質問を使用して、PoC (概念実証) の目標を確認できます。

  • データウェアハウスをスケーリングする目標は何ですか。

  • 条件を改善したい特定のサービスレベルアグリーメントはどれですか?

  • お客様がデータウェアハウスに含める必要がある新しいデータセットは何ですか?

  • テストおよび測定する必要があるビジネスクリティカルな SQL クエリはどれですか? さまざまなクエリのタイプ (例: 選択、挿入、更新、削除) など、完全な範囲の SQL の複雑さを確実に含めます。

  • テストする予定のワークロードの全般的な種類は何ですか。この例として、ワークロードの抽出、変換、ロード (ETL)、クエリの報告、バッチ抽出などがあります。

これらの質問に答えると、概念実証を構築するための SMART 目標と成功基準を確立できるようになります。目標設定の詳細については、ウィキペディアの SMART 基準を参照してください。

概念実証の設定

オンプレミスのデータウェアハウスからハードウェアのプロビジョニング、ネットワーキング、ソフトウェアのインストールが不要になったため、独自のデータセットを使用して Amazon Redshift を試すことがかつてないほど簡単になりました。以前必要だったサイジングの決定や見積もりの多くは、クリックするだけで済みます。クラスターのサイズを柔軟に変更したり、ストレージとコンピューティングの比率を調整したりできます。

簡単に言うと、Amazon Redshift の概念実証環境を設定するには、2 つのステップがあります。これには、データウェアハウスの起動と、評価のためのスキーマとデータセットの変換が含まれます。

起動クラスターのサイズを選択します。

Amazon Redshift コンソールを使用して、ノードタイプと数を選択できます。概念実証計画の一環として、クラスターのサイズ変更もテストすることをお勧めします。クラスターの初期サイズを取得するには、次のステップを実行します。

  1. AWS Management Console にサインインして https://console.aws.amazon.com/redshift/で Amazon Redshift コンソールを開きます。

  2. ナビゲーションメニューで、[Create cluster] (クラスターの作成) を選択して構成ページを開きます。

  3. [Cluster identifier] (クラスター識別子) で、クラスターの名前を入力します。

  4. 以下のステップでは、RA3 ノードタイプをサポートする AWS リージョンで実行されている、Amazon Redshift コンソールについて説明します。RA3 ノードタイプをサポートする AWS リージョンのリストについては、「Amazon Redshift 管理ガイド」の「RA3 ノードタイプの概要」を参照してください。

    クラスターのサイズがわからない場合は、[Help me choose (選択ヘルプ)] を選んでください。これにより、データウェアハウスに保存する予定のデータのサイズとクエリの特性について質問するサイジング計算ツールが起動されます。クラスターの必須サイズ (ノードタイプとノード数) がわかっている場合は、[I'll choose (選択する)] を選んでください。次に、ノードの種類とノードの数量を選択して、概念実証のためにクラスターのサイズを設定します。

    注記

    組織が適格であり、Amazon Redshift Serverless が利用できない AWS リージョン でクラスターが作成されている場合、Amazon Redshift 無料トライアルプログラムでクラスターを作成できる場合があります。[このクラスターを何に使用する予定ですか?] という質問に対して、[本番稼働用] または [無料トライアル] のいずれかを選択します。[無料トライアル] を選択したときには、dc2.large ノードタイプの設定を作成します。無料トライアルの選択に関する詳細については、「Amazon Redshift 無料トライアル」を参照してください。Amazon Redshift Serverless が利用可能な AWS リージョン の一覧については、「Amazon Web Services 全般のリファレンス」の「Redshift Serverless API」に記載されているエンドポイントを参照してください。

  5. 必要なすべてのクラスタープロパティを入力したら、[Create cluster] (クラスターの作成) を選択してデータウェアハウスを起動します。

    Amazon Redshift コンソールを使用したクラスター作成の詳細については、「Amazon Redshift 管理ガイド」の「クラスターの作成」を参照してください。

スキーマを変換し、概念実証のためのデータセットを設定します。

既存のデータウェアハウスをお持ちでない場合は、このセクションをスキップして、Amazon Redshift 入門ガイドをご覧ください。Amazon Redshift 入門ガイドでは、クラスターの作成に関するチュートリアルと、Amazon Redshift でのデータ設定の例を示しています。

既存のデータウェアハウスからの移行の際には、AWS Schema Conversion Tool および AWS Database Migration Service を使用して、スキーマ、コード、およびデータを変換できます。選択するツールは、データのソースと、オプションで継続するレプリケーションによって異なります。詳細については、AWSSchema Conversion Tool ユーザーガイドの「AWS Schema Conversion Toolとは」、ならびに、AWS Database Migration Serviceユーザーガイドの「AWS Database Migration Service とは」を参照してください。Amazon Redshift のデータの設定には、以下が役立ちます。

  • AWS Schema Conversion Tool を使用した Amazon Redshift へのデータウェアハウスの移行 – このブログ記事では、AWS SCT のデータ抽出機能を使用して、既存のデータウェアハウスを Amazon Redshift に移行する方法の概要を示します。AWS SCT ツールでは、多くのレガシープラットフォーム (Oracle、Greenplum、Netezza、Teradata、Microsoft SQL Server、Vertica など) からの、データの移行が可能です。

  • オプションとして、ソースで変更されたデータの継続的なレプリケーションに、AWS Database Migration Service を使用することもできます。詳細については、AWS Database Migration Service ユーザーガイドの「AWS Database Migration Service のターゲットとしての Amazon Redshift データベースの使用」を参照してください。

Amazon Redshift はリレーショナルデータベース管理システム (RDBMS) です。そのため、star スキーマ、snowflake スキーマ、データ vault モデル、シンプルテーブル、フラットテーブル、または正規化されたテーブルなど、さまざまなタイプのデータモデルを実行できます。Amazon Redshift でスキーマを設定した後は、超並列処理と列指向データストレージを利用して、すぐに使用できる高速な分析クエリを実行できます。スキーマタイプの詳細については、ウィキペディアから「star スキーマ」、「snowflake スキーマ」、および「データ Vault のモデリング」を参照してください。

完全な評価のためのチェックリスト

完全な評価がすべてのデータウェアハウスのニーズを満たしていることを確認します。成功基準には、次の項目を含めることを検討してください。

  • データのロード時間 – COPY コマンドの使用は、データをロードするためにかかる時間をテストするための一般的な方法です。詳細については、「データをロードするための Amazon Redshift のベストプラクティス」を参照してください。

  • クラスターのスループット – 1 時間あたりのクエリの測定は、スループットを決定するための一般的な方法です。そのためには、ワークロードの一般的なクエリを実行するテストを設定します。

  • データセキュリティ — Amazon Redshift では、保管時のデータと転送中のデータを簡単に暗号化できます。また、キーを管理するための数多くのオプションがあります。Amazon Redshift はシングルサインオンの統合もサポートしています。Amazon Redshift の料金には、組み込みのセキュリティ、データ圧縮、バックアップストレージ、データ転送が含まれます。

  • サードパーティー製ツールの統合 – JDBC または ODBC 接続を使用して、ビジネスインテリジェンスツールや他の外部ツールと統合できます。

  • 他の AWS サービスとの相互運用性 – Amazon Redshift は、他の AWS サービス (Amazon EMR、Amazon QuickSight、AWS Glue、Amazon S3、Amazon Kinesis など) と統合されています。この統合は、データウェアハウスの設定と管理で使用できます。

  • バックアップとスナップショット – バックアップおよびスナップショットが自動的に作成されます。ポイントインタイムのスナップショットは、いつでも作成することも、スケジュールに基づいて作成することもできます。評価の一環として、スナップショットの使用と 2 番目のクラスターの作成を試します。開発およびテスト組織がクラスターを使用できるかどうかテストします。

  • サイズ変更 — Amazon Redshift ノードの数またはタイプの増加を含める必要があります。サイズ変更の前後のワークロードスループットが、ワークロードボリュームのばらつきに対応しているかどうかを評価します。詳細については、「Amazon Redshift 管理ガイド」の「Amazon Redshift クラスターのサイズ調整」を参照してください。

  • 同時実行スケーリング – この特徴を使用すれば、データウェアハウス内のトラフィック量の変動を処理できます。同時実行スケーリングを使用すると、一貫した高速のクエリパフォーマンスで、事実上無制限の同時ユーザーと同時クエリをサポートできます。詳細については、「同時実行スケーリングを使用する」を参照してください。

  • 自動ワークロード管理 (WLM) – 自動 WLM を使用して、ビジネスクリティカルなクエリを他のクエリよりも優先します。ワークロードに基づいてキューを設定してみてください (ETL 用のキューとレポート用のキューなど)。次に、自動 WLM を有効にして、同時実行性とメモリリソースを動的に割り当てます。詳細については、「自動 WLM の実装」を参照してください。

  • Amazon Redshift Advisor – Advisor は、カスタマイズされた推奨事項を作成して、クラスターのワークロードと使用状況メトリクスを分析することで、パフォーマンスを向上させ、コストを最適化します。Amazon Redshift コンソールで Advisor レコメンデーションを表示するには、登録してください。詳細については、「Amazon Redshift Advisor からのレコメンデーションの使用」を参照してください。

  • テーブル設計 – Amazon Redshift は、ほとんどのワークロードですぐに使用できる優れたパフォーマンスを提供します。テーブルを作成する場合、デフォルトのソートキーとディストリビューションキーは AUTO です。詳細については、「自動テーブル最適化の使用」を参照してください。

  • サポート – 評価の対象として、AWS Support も含めることを強くお勧めします。また、概念実証の際には、アカウントマネージャーにお問い合わせいただくことも推奨します。AWS は対象となるお客様に対して、概念実証のための技術ガイダンスとクレジットによる支援を提供しています。お探しのサポートが見つからない場合は、Amazon Redshift チームに直接ご相談ください。ヘルプが必要な場合は、Amazon Redshift の概念実証のサポートをリクエストするからフォームを送信してください。

  • レイクハウスの統合 – 組み込みの統合機能を使用して、すぐに使用できる Amazon Redshift Spectrum 機能をお試しください。Redshift Spectrum を使用すれば、データウェアハウスをデータレイクに拡張し、既存のクラスターを使用して Amazon S3 のペタバイト規模のデータに対してクエリを実行できます。詳細については、「Amazon Redshift Spectrum を使用した外部データのクエリ」を参照してください。

評価のためのプロジェクト計画を作成する

クエリベンチマークを作成するための次の手法の一部は、Amazon Redshift の評価をサポートする場合に役に立ちます。

  • 各ランタイムカテゴリのクエリの一覧を収集します。十分な数 (たとえば、カテゴリあたり 30) を持つことで、現実的なデータウェアハウスの実装が評価に反映されていることを確認できます。評価に含める各クエリを、評価に対して確立するいずれかのカテゴリに関連付けるための一意の識別子を追加します。次に、これらの一意の識別子を使用して、システムテーブルからスループットを決定できます。

    また、クエリグループを作成して、評価クエリを整理することができます。たとえば、評価の「Reporting」カテゴリを確立した場合、評価クエリに「Report」という単語のタグを付けるコーディングシステムを作成できます。これにより、レポート内の個別のクエリを R1、R2 のように識別できます。次の例で、この方法を示します。

    
SELECT 'Reporting' AS query_category, 'R1' as query_id, * FROM customers;              

    
SELECT query, datediff(seconds, starttime, endtime)
  FROM stl_query 
  WHERE 
  querytxt LIKE '%Reporting%' 
  and starttime >= '2018-04-15 00:00' 
  and endtime < '2018-04-15 23:59';

    クエリを評価カテゴリと関連付けた場合、一意の識別子を使用して、各カテゴリのシステムテーブルからのスループットを判断できます。

  • 既存のデータウェアハウスで、さまざまなランタイム時間がある履歴のユーザークエリまたは ETL クエリを使用してスループットをテストします。オープンソースの JMeter やカスタムユーティリティなどの負荷テストユーティリティを使用する場合があります。その場合は、ユーティリティが次の操作を実行していることを確認してください。

    • ネットワーク送信時間を考慮に入れることができます。

    • 内部システムテーブルのスループットに基づいて実行時間を評価します。これを行う方法については、「クエリの概要の分析」を参照してください。

  • 評価中にテストする予定のさまざまな変形をすべて識別します。次のリストに、一般的ないくつかの変数を示します。

    • クラスターサイズ

    • ノードの種類

    • ロードテストの期間

    • 同時実行の設定

  • 営業時間外および週末にクラスターを一時停止することで、概念実証のコストを削減できます。クラスターが一時停止すると、オンデマンドコンピューティングの請求は一時停止されます。クラスターでテストを実行するには、秒単位の請求を再開します。スケジュールを作成して、クラスターを自動的に一時停止および再開することもできます。詳細については、「Amazon Redshift 管理ガイド」の「クラスターの一時停止と再開」を参照してください。

この段階で、プロジェクト計画を完了し、結果を評価する準備が整いました。
