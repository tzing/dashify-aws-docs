Melakukan pembuktian konsep untuk Amazon Redshift

Amazon Redshift adalah data warehouse dengan skala yang cepat, dan membuatnya sederhana dan hemat biaya untuk menganalisis semua data Anda dengan menggunakan alat business intelligence Anda yang sudah ada. Amazon Redshift menawarkan kinerja cepat di gudang data cloud berbiaya rendah. Ini menggunakan optimasi permintaan canggih, cache dipercepat, penyimpanan kolumnar pada disk lokal kinerja tinggi, dan eksekusi permintaan parallel secara besar-besaran.

Di bagian berikutnya, Anda dapat menemukan kerangka kerja untuk membangun pembuktian konsep dengan Amazon Redshift. Kerangka kerja ini membantu Anda menggunakan praktik terbaik arsitektur untuk merancang dan mengoperasikan gudang data yang aman, berkinerja tinggi, dan hemat biaya. Panduan ini didasarkan pada meninjau desain ribuan arsitektur pelanggan di berbagai jenis bisnis dan kasus penggunaan. Kami telah mengumpulkan pengalaman pelanggan untuk mengembangkan serangkaian praktik terbaik ini untuk membantu Anda mengembangkan kriteria untuk mengevaluasi beban kerja gudang data Anda.

Ikhtisar prosesnya

Melakukan pembuktian konsep adalah proses tiga langkah:

  1. Identifikasi tujuan bukti konsep - Anda dapat bekerja mundur dari persyaratan bisnis dan kriteria keberhasilan Anda, dan menerjemahkannya ke dalam bukti teknis rencana proyek konsep.

  2. Siapkan bukti lingkungan konsep - sebagian besar proses penyiapan adalah klik beberapa tombol untuk membuat sumber daya Anda. Dalam beberapa menit, Anda dapat memiliki lingkungan gudang data yang siap dengan data dimuat.

  3. Lengkapi bukti rencana proyek konsep untuk memastikan bahwa tujuan terpenuhi.

Pada bagian berikut, kita masuk ke detail setiap langkah.

Identifikasi tujuan bisnis dan kriteria keberhasilan

Mengidentifikasi tujuan bukti konsep memainkan peran penting dalam menentukan apa yang ingin Anda ukur sebagai bagian dari proses evaluasi. Kriteria evaluasi harus mencakup tantangan penskalaan saat ini, penyempurnaan untuk meningkatkan pengalaman pelanggan Anda tentang gudang data, dan metode mengatasi titik nyeri operasional Anda saat ini. Anda dapat menggunakan pertanyaan-pertanyaan berikut untuk mengidentifikasi tujuan bukti konsep:

  • Apa sasaran Anda untuk menskalakan data Anda?

  • Apa perjanjian tingkat layanan khusus yang persyaratannya ingin Anda tingkatkan?

  • Kumpulan data baru apa yang perlu Anda sertakan di gudang data Anda?

  • Apa kueri SQL penting bisnis yang perlu Anda uji dan ukur? Pastikan untuk menyertakan berbagai kompleksitas SQL, seperti berbagai jenis kueri (misalnya, pilih, masukkan, perbarui, dan hapus).

  • Apa jenis beban kerja umum yang Anda rencanakan untuk diuji? Contohnya mungkin termasuk beban kerja extract-transform-load (ETL), kueri pelaporan, dan ekstrak batch.

Setelah Anda menjawab pertanyaan-pertanyaan ini, Anda harus dapat menetapkan tujuan SMART dan kriteria keberhasilan untuk membangun bukti konsep Anda. Untuk informasi tentang menetapkan tujuan, lihat kriteria SMART di Wikipedia.

Siapkan bukti konsep Anda

Karena kami menghilangkan penyediaan perangkat keras, jaringan, dan instalasi perangkat lunak dari gudang data lokal, mencoba Amazon Redshift dengan set data Anda sendiri tidak pernah semudah ini. Banyak keputusan ukuran dan estimasi yang dulu diperlukan sekarang hanya dengan sekali klik. Anda dapat mengubah ukuran klaster secara fleksibel atau menyesuaikan rasio penyimpanan versus komputasi.

Secara umum, menyiapkan bukti lingkungan konsep Amazon Redshift. Ini melibatkan peluncuran data warehouse dan kemudian konversi skema dan dataset untuk evaluasi.

Pilih ukuran klaster awal

Anda dapat memilih jenis node dan jumlah node menggunakan konsol Amazon Redshift. Kami menyarankan Anda juga menguji ukuran cluster sebagai bagian dari bukti rencana konsep Anda. Untuk mendapatkan ukuran awal untuk klaster Anda, lakukan langkah-langkah berikut:

  1. Masuklah ke AWS Management Console dan buka konsol Amazon Redshift di https://console.aws.amazon.com/redshift/.

  2. Di menu navigasi, pilih Buat klaster untuk membuka halaman konfigurasi.

  3. Untuk Pengenal klaster, masukkan nama untuk klaster.

  4. Langkah berikut menjelaskan konsol Amazon Redshift yang berjalan diAWS Wilayah yang mendukung jenis node RA3. Untuk daftarAWS Wilayah yang mendukung jenis node RA3, lihat Ikhtisar jenis node RA3 dalam Panduan Manajemen Amazon Redshift.

    Jika Anda tidak tahu seberapa besar ukuran cluster Anda, pilih Bantu saya memilih. Melakukan hal ini memulai kalkulator ukuran yang mengajukan pertanyaan tentang ukuran dan karakteristik kueri data yang Anda rencanakan untuk disimpan di gudang data Anda. Jika Anda tahu ukuran yang diperlukan cluster Anda (yaitu, jenis node dan jumlah node), pilih saya akan memilih. Kemudian pilih tipe Node dan jumlah Nodes untuk mengukur klaster Anda untuk bukti konsep.

Mengkonversi skema dan mengatur dataset untuk bukti konsep

Jika Anda tidak memiliki gudang data yang ada, lewati bagian ini dan lihat Panduan Memulai Amazon Redshift. Panduan Memulai Amazon Redshift menyediakan tutorial untuk membuat klaster dan contoh pengaturan data di Amazon Redshift.

Saat bermigrasi dari gudang data yang ada, Anda dapat mengonversi skema, kode, dan data menggunakanAWS Schema Conversion Tool danAWS Database Migration Service. Alat pilihan alat Anda tergantung pada sumber data Anda dan replikasi opsional. Untuk informasi selengkapnya, lihat Apa ituAWS Schema Conversion Tool? dalam Panduan PenggunaAWS Schema Conversion Tool dan Apa ItuAWS Database Migration Service? dalam AWS Database Migration ServiceUser Guide. Berikut ini dapat membantu Anda mengatur data di Amazon Redshift:

  • Migrasikan Gudang Data Anda ke Amazon Redshift MenggunakanAWS Schema Conversion Tool - posting blog ini memberikan ikhtisar tentang bagaimana Anda dapat menggunakan ekstraktorAWS SCT data untuk memigrasi gudang data yang ada ke Amazon Redshift. AWS SCTAlat ini dapat memigrasi data Anda dari banyak platform lama (seperti Oracle, Greenplum, Netezza, Teradata, Microsoft SQL Server, atau Vertica).

  • Secara opsional, Anda juga dapat menggunakanAWS Database Migration Service untuk replikasi berkelanjutan dari data yang diubah dari sumber. Untuk informasi selengkapnya, lihat Menggunakan Database Amazon Redshift sebagai TargetAWS Database Migration Service dalam PanduanAWS Database Migration Service Pengguna.

Amazon Redshift adalah sistem manajemen relasional (RDBMS). Dengan demikian, dapat menjalankan berbagai jenis model data termasuk skema bintang, skema kepingan salju, model kubah data, dan tabel sederhana, datar, atau dinormalisasi. Setelah menyiapkan skema Anda di Amazon Redshift, Anda dapat memanfaatkan pemrosesan parallel secara masif dan penyimpanan data kolumnar untuk kueri analitik cepat di luar kotak. Untuk informasi tentang jenis skema, lihat skema bintang, skema kepingan salju, dan pemodelan kubah data di Wikipedia.

Checklist untuk evaluasi lengkap

Pastikan evaluasi lengkap memenuhi semua kebutuhan gudang data Anda. Pertimbangkan untuk memasukkan item berikut dalam kriteria kesuksesan Anda:

  • Waktu buka data - menggunakanCOPY perintah adalah cara umum untuk menguji berapa lama waktu yang dibutuhkan untuk memuat data. Untuk informasi selengkapnya, lihat Praktik terbaik Amazon Redshift untuk memuat data.

  • Throughput klaster — mengukur kueri per jam adalah cara umum untuk menentukan throughput. Untuk melakukannya, siapkan pengujian untuk menjalankan kueri khas untuk beban kerja Anda.

  • Keamanan data — Anda dapat dengan mudah mengenkripsi data at rest dan transit dengan Amazon Redshift. Anda juga memiliki sejumlah opsi untuk mengelola kunci. Amazon Redshift juga mendukung integrasi sistem masuk tunggal. Harga Amazon Redshift mencakup keamanan bawaan, kompresi data, penyimpanan cadangan, dan transfer data.

  • Integrasi alat pihak ketiga - Anda dapat menggunakan koneksi JDBC atau ODBC untuk berintegrasi dengan intelijen bisnis dan alat eksternal lainnya.

  • Interoperabilitas denganAWS layanan lainnya — Amazon Redshift berintegrasi denganAWS layanan lainnya Amazon QuickSight S3 seperti Amazon Kinesis.AWS Glue Anda dapat menggunakan integrasi ini saat menyiapkan dan mengelola gudang data Anda.

  • Backup dan snapshot — backup dan snapshot dibuat secara otomatis. Anda juga dapat membuat point-in-time snapshot kapan saja atau sesuai jadwal. Coba gunakan snapshot dan buat klaster kedua sebagai bagian dari evaluasi Anda. Evaluasi apakah organisasi pengembangan dan pengujian Anda dapat menggunakan klaster.

  • Mengubah ukuran — evaluasi Anda harus mencakup peningkatan jumlah atau jenis node Amazon Redshift. Evaluasi bahwa throughput beban kerja sebelum dan sesudah pengubahan ukuran memenuhi variabilitas volume beban kerja Anda. Untuk informasi selengkapnya, lihat Mengubah Ukuran klaster di Amazon Redshift di Panduan Manajemen Amazon Redshift.

  • Penskalaan konkurensi - fitur ini membantu Anda menangani variabilitas volume lalu lintas di gudang data Anda. Dengan penskalaan konkurensi, Anda dapat mendukung pengguna bersamaan yang hampir tidak terbatas dan kueri bersamaan, dengan kinerja kueri yang cepat secara konsisten. Untuk informasi selengkapnya, lihat Bekerja dengan penskalaan konkurensi.

  • Manajemen beban kerja otomatis (WLM) - memprioritaskan kueri penting bisnis Anda daripada kueri lain dengan menggunakan WLM otomatis. Coba siapkan antrian berdasarkan beban kerja Anda (misalnya, antrean untuk ETL dan antrean untuk pelaporan). Kemudian aktifkan WLM otomatis untuk mengalokasikan sumber daya konkurensi dan memori secara dinamis. Untuk informasi selengkapnya, lihat Menerapkan WLM otomatis.

  • Amazon Redshift Advisor — Penasihat mengembangkan rekomendasi khusus untuk meningkatkan kinerja dan mengoptimalkan biaya dengan menganalisis beban kerja dan metrik penggunaan untuk klaster Anda. Masuk ke Konsol Amazon Redshift untuk melihat rekomendasi Penasihat. Untuk informasi selengkapnya, lihat Bekerja dengan rekomendasi dari Amazon Redshift Advisor.

  • Desain meja - Amazon Redshift memberikan kinerja luar biasa di luar kotak untuk sebagian besar beban kerja. Saat Anda membuat tabel, kunci sortir default dan kunci distribusi adalahAUTO. Untuk informasi selengkapnya, lihat Bekerja dengan optimasi tabel otomatis.

  • Support - kami sangat menyarankan agar Anda mengevaluasiAWS Support sebagai bagian dari evaluasi Anda. Juga, pastikan untuk berbicara dengan manajer akun Anda tentang bukti konsep Anda. AWSdapat membantu dengan bimbingan teknis dan kredit untuk bukti konsep jika Anda memenuhi syarat. Jika Anda tidak menemukan bantuan yang Anda cari, Anda dapat berbicara langsung dengan tim Amazon Redshift. Untuk bantuan, kirimkan formulir di Minta dukungan untuk Amazon Redshift Anda proof-of-concept.

  • Integrasi rumah danau — dengan integrasi bawaan, coba gunakan fitur out-of-box Amazon Redshift Spectrum. Dengan Redshift Spectrum, Anda dapat memperluas gudang data ke data lake dan menjalankan kueri terhadap petabyte data di Amazon S3 menggunakan klaster yang ada. Untuk informasi selengkapnya, lihat Menanyakan data eksternal menggunakan Amazon Redshift Spectrum.

Kembangkan rencana proyek untuk evaluasi Anda

Beberapa teknik berikut untuk membuat tolok ukur kueri mungkin membantu mendukung evaluasi Amazon Redshift Anda:

  • Kumpulkan daftar kueri untuk setiap kategori runtime. Memiliki jumlah yang cukup (misalnya, 30 per kategori) membantu memastikan bahwa evaluasi Anda mencerminkan implementasi gudang data dunia nyata. Tambahkan pengenal unik untuk mengaitkan setiap kueri yang Anda sertakan dalam evaluasi dengan salah satu kategori yang Anda buat untuk evaluasi Anda. Anda kemudian dapat menggunakan pengidentifikasi unik ini untuk menentukan throughput dari tabel sistem.

    Anda juga dapat membuat grup kueri untuk mengatur kueri evaluasi Anda. Misalnya, jika Anda telah menetapkan kategori “Pelaporan” untuk evaluasi, Anda dapat membuat sistem pengkodean untuk menandai kueri evaluasi Anda dengan kata “Laporkan.” Anda kemudian dapat mengidentifikasi kueri individu dalam pelaporan sebagai R1, R2, dan sebagainya. Contoh berikut menunjukkan pendekatan ini.

    
SELECT 'Reporting' AS query_category, 'R1' as query_id, * FROM customers;              

    
SELECT query, datediff(seconds, starttime, endtime)
  FROM stl_query 
  WHERE 
  querytxt LIKE '%Reporting%' 
  and starttime >= '2018-04-15 00:00' 
  and endtime < '2018-04-15 23:59';

    Bila Anda telah menghubungkan kueri dengan kategori evaluasi, Anda dapat menggunakan pengenal unik untuk menentukan throughput dari tabel sistem untuk setiap kategori.

  • Uji throughput dengan pengguna historis atau kueri ETL yang memiliki berbagai runtime di gudang data Anda yang ada. Anda mungkin menggunakan utilitas pengujian beban, seperti JMeter open-source atau utilitas kustom. Jika demikian, pastikan bahwa utilitas Anda melakukan hal berikut:

    • Hal ini dapat mengambil waktu transmisi jaringan ke rekening.

    • Ini mengevaluasi waktu eksekusi berdasarkan throughput dari tabel sistem internal. Untuk informasi tentang cara melakukan ini, lihat Menganalisis ringkasan kueri.

  • Identifikasi semua berbagai permutasi yang Anda rencanakan untuk diuji selama evaluasi Anda. Daftar berikut menyediakan beberapa variabel umum:

    • ukuran cluster

    • Tipe simpul

    • Durasi pengujian beban

    • Pengaturan konkurensi

  • Kurangi biaya bukti konsep Anda dengan menjeda klaster Anda selama jam kerja dan akhir pekan. Saat klaster dijeda, penagihan komputasi sesuai permintaan akan ditangguhkan. Untuk menjalankan pengujian pada klaster, lanjutkan penagihan per detik. Anda juga dapat membuat jadwal untuk menjeda dan melanjutkan klaster Anda secara otomatis. Untuk informasi selengkapnya, lihat Menjeda dan melanjutkan kluster di Panduan Manajemen Amazon Redshift.

Pada tahap ini, Anda siap untuk menyelesaikan rencana proyek Anda dan mengevaluasi hasil.
