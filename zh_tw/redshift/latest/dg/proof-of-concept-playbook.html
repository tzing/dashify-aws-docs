執行 Amazon Redshift 的概念驗證

Amazon Redshift 是一項快速的可擴展資料倉儲服務，可讓您搭配現有商業智慧 (BI) 工具使用標準 SQL 來分析所有資料，操作簡單且符合成本效益。Amazon Redshift 可在低成本的雲端資料倉儲中提供快速的效能。其使用精密的查詢最佳化、加速快取、高效能本機磁碟上的直欄式儲存，以及大量執行平行查詢來達成此目的。

在以下幾節中，您可以找到使用 Amazon Redshift 建立概念驗證的框架。此框架有助於用來設計和操作安全、高效能且符合成本效益的資料倉儲的架構性最佳實務。此指導方針為檢閱了廣泛的業務類型和使用案例中數千個客戶架構的設計所得到的成果。我們集結了客戶體驗來開發這組最佳實務，以幫助您開發用於評估您的資料倉儲工作負載的條件。

程序概觀

執行概念驗證有三個步驟：

  1. 確定概念驗證的目標 – 您可以從業務需求和成功標準開始逆向進行，並將其轉換為技術性概念驗證專案計劃。

  2. 設定概念驗證環境 – 大部分的設定程序只要按一下幾個按鈕，即可建立您的資源。在幾分鐘內，您就可以準備好已載入資料的資料倉儲環境。

  3. 完成概念驗證專案計劃，以確保達到目標。

在以下各節中，我們會探索每個步驟的詳細資訊。

確定業務目標和成功標準

識別概念驗證的目標在判斷評估程序中所要測量的內容方面扮演著關鍵的角色。評估條件應該包括目前的擴展挑戰、用來改善資料倉儲客戶體驗的增強項目，以及解決您目前操作困境的方法。您可以使用下列問題來識別概念驗證的目標：

  • 擴展您資料倉儲的目標為何？

  • 您要改善其條款的特定服務水準協議為何？

  • 您需要在資料倉儲中包括什麼新資料集？

  • 您必須進行測試和衡量的商業關鍵 SQL 查詢為何？ 務必包括完整範圍的 SQL 複雜性，例如不同的查詢類型 (例如，選取、插入、更新和刪除)。

  • 您計畫測試的一般類型工作負載為何？ 範例包含擷取-轉換-載入 (ETL) 工作負載、報表查詢和批次擷取。

回答這些問題之後，您應該可以為建立您的概念驗證建立 SMART 目標和成功標準。有關設定目標的資訊，請參閱維基百科中的 SMART 準則。

設定您的概念驗證

由於我們免除了內部部署資料倉儲的硬體佈建、網路和軟體安裝，因此使用自己的資料集嘗試使用 Amazon Redshift 從未如此簡單。許多過去需要的大小瘸測和估計現在只需單擊一下即可。您可以彈性調整叢集的大小，或調整儲存空間與運算的比例。

設定 Amazon Redshift 概念驗證環境需要兩個步驟。其中涉及資料倉儲的啟動，然後用於評估的結構描述和資料集轉換。

選擇起始叢集大小

您可以使用 Amazon Redshift 主控台選擇節點類型和節點數目。建議您也在概念驗證計劃過程中測試調整叢集大小。若要取得叢集的初始大小，請執行下列步驟：

  1. 登入 AWS Management Console，開啟位於 https://console.aws.amazon.com/redshift/ 的 Amazon Redshift 主控台。

  2. 在瀏覽功能表上，選擇建立叢集以開啟組態頁面。

  3. 對於叢集叢集識別符，輸入您叢集的名稱。

  4. 下列步驟說明在支援 RA3 節點類型的 AWS 區域中執行的 Amazon Redshift 主控台。如需支援 RA3 節點類型的 AWS 區域清單，請參閱《Amazon Redshift 管理指南》中的 RA3 節點類型概觀。

    如果您不知道要將叢集的大小調整到多大，請選擇協助我選擇。這樣做會啟動調整大小計算器，詢問您有關計劃儲存在資料倉儲中之資料的大小和查詢特性的問題。如果您知道叢集的所需大小 (也就是節點類型和節點數目)，請選擇我會選擇。然後選擇節點類型和節點數量來調整叢集的大小以進行概念驗證。

    注意

    如果您的組織符合資格，而且您的叢集是在未提供 Amazon Redshift Serverless 的 AWS 區域中建立，您也許可以在 Amazon Redshift 免費試用方案下建立叢集。選擇生產或免費試用來回答問題您打算將此叢集用於什麼目的？ 選擇免費試用時，您可以使用 dc2.large 節點類型建立組態。如需有關選擇免費試用的詳細資訊，請參閱 Amazon Redshift 免費試用版。如需有提供 Amazon Redshift Serverless 的 AWS 區域 清單，請參閱Amazon Web Services 一般參考中針對 Redshift Serverless API 所列出的端點。

  5. 輸入所有必要的叢集屬性後，請選擇建立叢集以啟動資料倉儲。

    如需使用 Amazon Redshift 主控台建立叢集的詳細資訊，請參閱《Amazon Redshift 管理指南》中的建立叢集。

轉換結構描述並設定資料集以進行概念驗證

如果您沒有現有的資料倉儲，請略過本節並參閱《Amazon Redshift 入門指南》https://docs.aws.amazon.com/redshift/latest/gsg/。Amazon Redshift 入門指南會提供建立叢集的教學課程，以及在 Amazon Redshift 中設定資料的範例。

從現有的資料倉儲移轉時，您可以使用 AWS Schema Conversion Tool 和 AWS Database Migration Service 來轉換結構描述、程式碼和資料。您選擇的工具取決於資料來源和可選擇的持續複寫。如需詳細資訊，請參閱《AWS Schema Conversion Tool 使用者指南》中的什麼是 AWS Schema Conversion Tool？，以及《AWS Database Migration Service 使用者指南》中的什麼是 AWS Database Migration Service?。下列資訊可幫助您在 Amazon Redshift 中設定您的資料：

  • 使用 AWS Schema Conversion Tool 將您的資料倉儲遷移至 Amazon Redshift – 此部落格貼文提供的概觀可讓您了解您能如何使用 AWS SCT 資料擷取器，將您的現有資料倉儲遷移至 Amazon Redshift。AWS SCT 工具可以從許多傳統平台遷移您的資料 (例如 Oracle、Greenplum、Netezza、Teradata、Microsoft SQL Server 或 Vertica)。

  • 或者，您也可以將 AWS Database Migration Service 用於來源中變更資料的持續複製。如需詳細資訊，請參閱《AWS Database Migration Service 使用者指南》中的使用 Amazon Redshift 資料庫做為 AWS Database Migration Service 的目標。

Amazon Redshift 是關聯式資料庫管理系統 (RDBMS)。因此可以執行許多類型的資料模型，包括星狀結構描述、雪花結構描述、資料儲存庫模型，以及簡單、平面或標準化資料表。在 Amazon Redshift 中設定結構描述後，您可以利用大量平行處理和單欄式資料儲存，進行現成的快速分析查詢。有關結構描述類型的資訊，請參閱維基百科中的星形結構描述、雪花結構描述和資料儲存庫建模。

完整評估的檢查清單

確保完整的評估符合您的所有資料倉儲需求。請考慮在成功標準中包含下列項目：

  • 資料載入時間 – 使用 COPY 命令是測試載入資料所花費時間的常見方式。如需更多詳細資訊，請參閱 載入資料的 Amazon Redshift 最佳實務。

  • 叢集的輸送量 – 每小時測量查詢是判斷輸送量的常見方式。若要這麼做，請設定測試來為您的工作負載執行一般的查詢。

  • 資料安全性 – 您可以輕鬆加密 Amazon Redshift 的靜態資料與傳輸中的資料。您也有許多管理金鑰的選項。Amazon Redshift 也支援單一登入整合。Amazon Redshift 定價包含內建安全性、資料壓縮、備份儲存和資料傳輸。

  • 第三方工具整合 – 您可以使用 JDBC 或 ODBC 連線來與商業智慧和其他外部工具整合。

  • 與其他 AWS 服務的互通性 – Amazon Redshift 能夠與其他 AWS 服務整合，例如 Amazon EMR、Amazon QuickSight、AWS Glue、Amazon S3 和 Amazon Kinesis。設定和管理您的資料倉儲時，您可以使用此整合。

  • 備份和快照 – 自動建立備份和快照。也能夠隨時或依排程來建立時間點快照。嘗試在您的評估中使用快照和建立第二個叢集。評估您的開發和測試組織是否可以使用叢集。

  • 規模調整 – 您的評估應該包括增加 Amazon Redshift 節點類型的數量或類型。評估調整大小之前和之後的工作負載輸送量是否符合工作負載量的任何變化性。如需詳細資訊，請參閱《Amazon Redshift 管理指南》中的調整 Amazon Redshift 中叢集的大小。

  • 並行擴展 – 此功能可協助您處理資料倉儲中流量的變化性。有了並行擴展，您可以藉由持續的快速查詢效能，支援幾乎無限的並行使用者和並行查詢。如需更多詳細資訊，請參閱 使用並行擴展。

  • 自動工作負載管理 (WLM) – 使用自動 WLM 將業務關鍵查詢的優先順序排定為高於其他查詢。嘗試根據您的工作負載設定佇列 (例如，ETL 的佇列和報告的佇列)。然後啟用自動 WLM 以動態配置並行和記憶體資源。如需更多詳細資訊，請參閱 實作自動 WLM。

  • Amazon Redshift Advisor – Advisor 會透過分析叢集的工作負載和使用指標，開發客製化的建議，以提高效能並最佳化成本。登入 Amazon Redshift 主控台以檢視 Advisor 建議 如需更多詳細資訊，請參閱 使用 Amazon Redshift Advisor 的建議。

  • 資料表設計 – Amazon Redshift 可為大多數工作負載提供現成的絕佳效能。當您建立資料表時，預設的排序索引鍵和分佈索引鍵為 AUTO。如需更多詳細資訊，請參閱 使用自動資料表最佳化。

  • 支援 – 我們強裂建議您在評估時評估 AWS Support。此外，請務必與您的客戶經理討論您的概念驗證。如果您符合資格，AWS 可協助提供概念驗證的技術指導和學分。如果找不到所需的協助，可以直接與 Amazon Redshift 團隊聯絡。如需協助，請在為您的 Amazon Redshift 概念驗證要求支援中提交表單。

  • 資料湖倉儲整合 – 透過內建整合，請嘗試使用現成的 Amazon Redshift Spectrum 功能。使用 Redshift Spectrum，您可以將資料倉儲擴展到資料湖中，並使用現有叢集對 Amazon S3 中的 PB 級資料執行查詢。如需更多詳細資訊，請參閱 使用 Amazon Redshift Spectrum 查詢外部資料。

為您的評估制定專案計劃

下列有關建立查詢基準測試的一些技巧可能有助於支援您的 Amazon Redshift 評估：

  • 對每個執行時間類別收集查詢的清單。有足夠的數量 (例如，每個類別 30 個) 有助於確保您的評估可反映實際的資料倉儲實作。新增唯一的識別符，將您在您的評估中包含的每個查詢與您的評估建立的類別建立關聯。然後您可以使用這些唯一的識別符來判斷來自系統資料表的輸送量。

    您也可以建立查詢群組來組織您的評估查詢。例如，如果您已為您的評估建立「報表」類別，則可以建立程式碼系統來以「報表」一詞標記您的評估查詢。然後您可以在報表內以 R1、R2 等等識別個別查詢。下列範例示範此方法。

    
SELECT 'Reporting' AS query_category, 'R1' as query_id, * FROM customers;              

    
SELECT query, datediff(seconds, starttime, endtime)
  FROM stl_query 
  WHERE 
  querytxt LIKE '%Reporting%' 
  and starttime >= '2018-04-15 00:00' 
  and endtime < '2018-04-15 23:59';

    將查詢與評估類別建立關聯之後，您可以使用唯一的識別符來判斷每個類別來自系統資料表的輸送量。

  • 測試歷史使用者或在您的現有資料倉儲中有許多執行期的 ETL 查詢的輸送量。您可以使用負載測試公用程式，例如開放原始碼的 JMeter 或自訂公用程式。若是如此，請確定您的公用程式執行下列事項：

    • 其可以將網路傳輸時間納入考量。

    • 會根據內部系統資料表輸送量來評估執行時間。如需如何進行該服務的詳細資訊，請參閱分析查詢摘要。

  • 識別您計畫在您的評估期間測試的各種排列。下列清單提供一些通用的變數：

    • 叢集大小

    • 節點類型

    • 負載測試期間

    • 並行設定

  • 在下班時間和週末暫停叢集，以降低概念驗證的成本。叢集暫停時，隨選運算計費就會暫停。若要在叢集上執行測試，請恢復為每秒計費。您也可以建立排程以自動暫停和恢復叢集。如需詳細資訊，請參閱《Amazon Redshift 管理指南》中的暫停和恢復叢集。

在這個階段中，您已經準備好完成專案計劃並評估結果。
