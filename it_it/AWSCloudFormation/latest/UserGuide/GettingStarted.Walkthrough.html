Inizia a usare

Con il modello giusto, puoi distribuire contemporaneamente tutte le AWS risorse necessarie per un'applicazione. In questa sezione esaminerai un modello che dichiara le risorse per un WordPress blog, crea un WordPress blog come pila, monitora il processo di creazione dello stack, esamina le risorse nello stack e quindi elimina lo stack. Utilizzate il per completare queste attività. AWS Management Console

Fase 1: seleziona un modello

Innanzitutto, è necessario un modello che specifichi le risorse che desideri nel tuo stack. Per questa fase è possibile utilizzare un modello di esempio già preparato. Il modello di esempio crea un WordPress blog di base che utilizza una singola istanza Amazon EC2 con un database MySQL locale per lo storage. Il modello crea inoltre un gruppo di sicurezza Amazon EC2 per controllare le impostazioni del firewall per l'istanza Amazon EC2.

Importante

AWS CloudFormation è gratuito, ma le AWS risorse che CloudFormation crea sono attive (e non in esecuzione in una sandbox). Ti sarà addebitato il costo di utilizzo standard per queste risorse finché non le termini nell'ultima attività di questo tutorial. L'addebito totale sarà minimo. Per informazioni su come ridurre al minimo i costi, consulta http://aws.amazon.com/free/.

Visualizzazione del modello

  • Puoi visualizzare il modello di esempio JSON o YAML WordPress . Non è necessario scaricarlo perché utilizzerai il modello URL più avanti in questa guida. Consulta Formati di modello AWS CloudFormation per ulteriori informazioni sui formati di modello.

Un modello è un file di testo JSON o YAML che contiene le informazioni di configurazione sulle AWS risorse che si desidera creare nello stack. Per questa guida, il modello di esempio include sei sezioni di primo livello: AWSTemplateFormatVersion, Description, Parameters, Mappings, Resources e Outputs; tuttavia, solo la sezione Resources è obbligatoria.

La sezione Risorse contiene le definizioni delle AWS risorse che desideri creare con il modello. Ogni risorsa è elencata separatamente e specifica le proprietà che sono necessarie per la creazione di una determinata risorsa. La seguente dichiarazione di risorse è la configurazione per l'istanza EC2, che in questo esempio ha il nome logico WebServer:

Esempio JSON
"Resources" : {
  ...    
  "WebServer": {
    "Type" : "AWS::EC2::Instance",
    "Properties": {
      "ImageId" : { "Fn::FindInMap" : [ "AWSRegionArch2AMI", { "Ref" : "AWS::Region" },
                        { "Fn::FindInMap" : [ "AWSInstanceType2Arch", { "Ref" : "InstanceType" }, "Arch" ] } ] },
      "InstanceType"   : { "Ref" : "InstanceType" },
      "SecurityGroups" : [ {"Ref" : "WebServerSecurityGroup"} ],
      "KeyName"        : { "Ref" : "KeyName" },
      "UserData" : { "Fn::Base64" : { "Fn::Join" : ["", [
                     "#!/bin/bash -xe\n",
                     "yum update -y aws-cfn-bootstrap\n",

                     "/opt/aws/bin/cfn-init -v ",
                     "         --stack ", { "Ref" : "AWS::StackName" },
                     "         --resource WebServer ",
                     "         --configsets wordpress_install ",
                     "         --region ", { "Ref" : "AWS::Region" }, "\n",

                     "/opt/aws/bin/cfn-signal -e $? ",
                     "         --stack ", { "Ref" : "AWS::StackName" },
                     "         --resource WebServer ",
                     "         --region ", { "Ref" : "AWS::Region" }, "\n"
      ]]}}
    },
    ...
  },
  ...  
  "WebServerSecurityGroup" : {
    "Type" : "AWS::EC2::SecurityGroup",
    "Properties" : {
      "GroupDescription" : "Enable HTTP access via port 80 locked down to the load balancer + SSH access",
      "SecurityGroupIngress" : [
        {"IpProtocol" : "tcp", "FromPort" : 80, "ToPort" : 80, "CidrIp" : "0.0.0.0/0"},

      ]
    }
  },
  ...

Esempio YAML
Resources: 
  ...    
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !FindInMap [AWSRegionArch2AMI, !Ref 'AWS::Region', !FindInMap [AWSInstanceType2Arch, !Ref InstanceType, Arch]]      
      InstanceType:
        Ref: InstanceType
      KeyName:
        Ref: KeyName
      SecurityGroups:
      - Ref: WebServerSecurityGroup
      UserData:
        Fn::Base64: !Sub |
           #!/bin/bash -xe
           yum update -y aws-cfn-bootstrap
           /opt/aws/bin/cfn-init -v --stack ${AWS::StackId} --resource WebServer --configsets wordpress_install --region ${AWS::Region}
           /opt/aws/bin/cfn-signal -e $? --stack ${AWS::StackId} --resource WebServer --region ${AWS::Region}
    ...
  ...
  
  WebServerSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: "Enable HTTP access via port 80 locked down to the load balancer + SSH access"
      SecurityGroupIngress:
      - CidrIp: 0.0.0.0/0
        FromPort: 80
        IpProtocol: tcp
        ToPort: 80
      - CidrIp: !Ref SSHLocation
        FromPort: 22
        IpProtocol: tcp
        ToPort: 22
  ...

Se hai creato istanze EC2 in precedenza puoi riconoscere proprietà, ad esempio ImageId, InstanceTypee KeyName, che determinano la configurazione dell'istanza. Le dichiarazioni di risorse sono un modo efficiente per specificare tutte queste impostazioni di configurazione in una sola volta. Quando si inseriscono dichiarazioni di risorse in un modello, è possibile creare e configurare tutte le risorse dichiarate utilizzando il modello per creare uno stack. Creare un nuovo stack che utilizza il medesimo modello per avviare la stessa configurazione delle risorse.

La dichiarazione di risorse inizia con una stringa che specifica il nome logico della risorsa. Come vedrai, il nome logico può essere utilizzato per fare riferimento alle risorse all'interno del modello.

Puoi utilizzare la sezione Parameters (Parametri) per dichiarare valori che è possibile passare al modello al momento della creazione dello stack. Un parametro è un modo efficace per specificare informazioni sensibili, ad esempio i nomi utenti e le password, che non si desidera archiviare nel modello stesso. È anche un modo per specificare le informazioni che potrebbero essere univoche per l'applicazione o configurazione specifica che si sta distribuendo, per esempio, un nome di dominio o un tipo di istanza. Quando crei lo WordPress stack più avanti in questa sezione, vedrai il set di parametri dichiarato nel modello apparire nella pagina Specificare i dettagli della procedura guidata Crea stack, dove puoi specificare i parametri prima di creare lo stack.

Nella dichiarazione di risorse WebServer, è possibile vedere la proprietà KeyName specificata con il parametro KeyName:

Esempio JSON
"WebServer" : {
  "Type": "AWS::EC2::Instance",
  "Properties": {
    "KeyName" : { "Ref" : "KeyName" },
    ...
  }
},
Esempio YAML
WebServer:
  Type: AWS::EC2::Instance
  Properties:
    KeyName:
      Ref: KeyName
    ...

Le parentesi graffe contengono una chiamata alla funzione Ref con KeyName come input. La funzione Ref restituisce il valore dell'oggetto cui fa riferimento. In questo caso, la funzione Ref imposta la proprietà KeyName sul valore specificato per KeyName quando è stato creato lo stack.

La funzione Ref può anche impostare la proprietà di una risorsa per il valore di un'altra risorsa. Ad esempio, la dichiarazione di risorse WebServer contiene la seguente dichiarazione di proprietà:

Esempio JSON
"WebServer" : {
  "Type": "AWS::EC2::Instance",
  "Properties": {
    ...
    "SecurityGroups" : [ {"Ref" : "WebServerSecurityGroup"} ],
    ...
  }
},
Esempio YAML
WebServer:
  Type: AWS::EC2::Instance
  Properties:
    SecurityGroups:
    - Ref: WebServerSecurityGroup
    ...

La proprietà SecurityGroups utilizza un elenco di gruppi di sicurezza di EC2. La funzione Ref dispone di un input di WebServerSecurityGroup, che è il nome logico di un gruppo di sicurezza nel modello e aggiunge il nome di WebServerSecurityGroup alla proprietà SecurityGroups.

Nel modello, è inoltre possibile trovare una sezione Mappings (Mappature). Puoi utilizzare le mappature per dichiarare i valori condizionali che vengono valutati in modo analogo a una tabella di ricerca. Il modello utilizza mappature per selezionare l'Amazon Machine Image (AMI) corretta per la Regione e il tipo di architettura per il tipo di istanza. Gli output definiscono valori personalizzati che vengono restituiti dal aws cloudformation describe-stacks comando e nella scheda Output della CloudFormation console dopo la creazione dello stack. È possibile utilizzare i valori di output per restituire informazioni dalle risorse nello stack, ad esempio l'URL di un sito Web che è stato creato nel modello. Le mappature, gli output e altri aspetti correlati ai modelli sono descritti in modo più dettagliato in Informazioni sulle nozioni di base dei modelli.

Fase 2: assicurati di avere preparato tutte le voci necessarie per lo stack

Prima di creare uno stack da un modello, è necessario assicurarsi che tutte le risorse dipendenti richieste dal modello siano disponibili. Un modello può utilizzare o fare riferimento sia a AWS risorse esistenti che a risorse dichiarate nel modello stesso. CloudFormation si occupa di controllare i riferimenti alle risorse nel modello e controlla anche i riferimenti alle risorse esistenti per assicurarsi che esistano nella regione in cui si sta creando lo stack. Se il modello si riferisce a una risorsa dipendente che non esiste, si verifica un errore di creazione dello stack.

Il WordPress modello di esempio contiene un parametro di input che specifica la coppia di chiavi utilizzata per l'istanza Amazon EC2 dichiarata nel modello. KeyName Il modello dipende dall'utente che crea uno stack dal modello per fornire una coppia di chiavi Amazon EC2 valida per il parametro KeyName. Se si fornisce un nome valido per la coppia di chiavi, lo stack viene creato senza problemi. Se non si fornisce un nome valido per la coppia di chiavi, viene eseguito il rollback dello stack.

Assicurati di disporre di una coppia di chiavi Amazon EC2 valida e registra il nome della coppia di chiavi prima di creare lo stack.

Per visualizzare le coppie di chiavi, apri la console Amazon EC2, quindi fare clic su Key Pairs (Coppie di chiavi) nel riquadro di navigazione.

Nota

Se non si dispone di una coppia di chiavi Amazon EC2 è necessario creare la coppia di chiavi nella stessa Regione in cui si crea lo stack. Per ulteriori informazioni sulla creazione di una coppia di chiavi, consulta la pagina relativa a come ottenere una coppia di chiavi SSH nella Guida per l'utente di Amazon EC2 per le istanze Linux.

Ora che hai una key pair valida, usiamo il WordPress modello per creare uno stack.

Fase 3: crea lo stack

Creerai il tuo stack in base al file WordPress-1.0.0 discusso in precedenza. Il modello contiene diverse AWS risorse, come un'istanza EC2.

Per creare lo stack WordPress

  1. Accedi AWS Management Console e apri la AWS CloudFormation console all'indirizzo https://console.aws.amazon.com/cloudformation.

  2. Scegli Crea stack.

  3. Nella sezione Specificare il modello, seleziona l'URL del modello Amazon S3 per digitare o incollare l'URL per il WordPress modello di esempio, quindi scegli Avanti:

    https://s3.us-west-2.amazonaws.com/cloudformation-templates-us-west-2/WordPress_Single_Instance.template

    Nota

    AWS CloudFormation i modelli archiviati in un bucket S3 devono essere accessibili all'utente che crea lo stack e devono trovarsi nella stessa regione dello stack che viene creato.

  4. Nella sezione Specifica i dettagli dello stack, immetti un nome nel campo Nome dello stack. Per questo esempio, utilizzare MyWPTestStack. Il nome dello stack non può contenere spazi.

  5. Nella sezione Parametri, è necessario fornire valori per tutti i parametri che non hanno valori predefiniti, inclusi dbUser, dbPassword, DB e. RootPassword KeyName Nel KeyNamecampo, inserisci il nome di una coppia Amazon EC2 valida nella stessa regione in cui stai creando lo stack.

  6. Seleziona Avanti.

  7. In questo scenario, non aggiungeremo alcun tag. Seleziona Avanti. I tag, che sono coppie chiave-valore, possono aiutarti a identificare i tuoi stack. Per ulteriori informazioni, consulta la pagina relativa all'aggiunta di tag allo stack CloudFormation .

  8. Rivedi le informazioni per lo stack. Quando si è soddisfatti delle impostazioni, seleziona Submit (Invia).

Per creare lo stack potrebbero essere necessari alcuni minuti, ma probabilmente non vorrai aspettare. Se sei come noi, vorrai sapere come sta andando la creazione dello stack.

Fase 4: monitora l'avanzamento della creazione dello stack

Dopo aver completato la procedura guidata Create Stack, CloudFormation inizia a creare le risorse specificate nel modello. Il nuovo stack viene visualizzato nell'elenco nella parte superiore della console. MyWPTestStack CloudFormation Il suo stato deve essere CREATE_IN_PROGRESS. È possibile vedere lo stato dettagliato di uno stack visualizzandone gli eventi.

Visualizzazione degli eventi per lo stack

  1. Sulla CloudFormation console, seleziona lo stack MyWPTestStack nell'elenco.

  2. Nel riquadro dei dettagli dello stack, scegliere la scheda Events (Eventi).

    La console aggiorna automaticamente l'elenco di eventi con gli eventi più recenti ogni 60 secondi.

La scheda Events (Eventi) mostra tutte le fasi principali nella creazione dello stack, ordinate per orario dell'evento, con gli ultimi eventi nella parte superiore.

Il primo evento (nella parte inferiore dell'elenco di eventi) è l'inizio del processo di creazione dello stack:

2013-04-24 18:54 UTC-7 CREATE_IN_PROGRESS AWS::CloudFormation::Stack MyWPTestStack User initiated

Di seguito vengono mostrati tutti gli eventi che segnano l'inizio e il completamento della creazione di ogni risorsa. Ad esempio, la creazione dell'istanza EC2 genera le seguenti voci:

2013-04-24 18:59 UTC-7 CREATE_COMPLETE AWS::EC2::Instance...

2013-04-24 18:54 UTC-7 CREATE_IN_PROGRESS AWS::EC2::Instance...

L'CREATE_IN_PROGRESSevento viene registrato quando CloudFormation segnala che ha iniziato a creare la risorsa. L'evento CREATE_COMPLETE viene registrato quando la risorsa viene creata senza errori.

Una volta CloudFormation creato con successo lo stack, vedrai il seguente evento nella parte superiore della scheda Eventi:

2013-04-24 19:17 UTC-7 CREATE_COMPLETE AWS::CloudFormation::Stack MyWPTestStack

Se non CloudFormation riesce a creare una risorsa, segnala un CREATE_FAILED evento e, per impostazione predefinita, ripristina lo stack ed elimina tutte le risorse che sono state create. La colonna Status Reason (Motivo dello stato) mostra il problema che ha causato l'errore.

Fase 5: utilizza le risorse del tuo stack

Quando lo stato dello stack MyWPTestStack è CREATE_COMPLETE, CloudFormation ha terminato la creazione dello stack ed è possibile iniziare a utilizzare le sue risorse.

Lo WordPress stack di esempio crea un sito Web. WordPress È possibile continuare con l' WordPress installazione eseguendo lo script di WordPress installazione.

Per completare l' WordPressinstallazione

  1. Nella scheda Outputs (Output), nella riga WebsiteURL (URL del sito Web), scegliere il collegamento nella colonna Value (Valore).

    Il valore di WebsiteURL output è l'URL dello script di installazione per il WordPress sito Web creato con lo stack.

  2. Nella pagina Web per l' WordPress installazione, segui le istruzioni visualizzate sullo schermo per completare l' WordPress installazione. Per ulteriori informazioni sull'installazione WordPress, vedere https://wordpress.org/support/article/how-to-install-wordpress/.

    Dopo aver completato l'installazione e aver effettuato l'accesso, verrai indirizzato alla dashboard in cui puoi impostare opzioni aggiuntive per il tuo WordPress blog. Successivamente, sarà possibile cominciare a scrivere post per il blog creato, utilizzando un modello CloudFormation.

Fase 6: pulizia

Hai completato le attività CloudFormation introduttive. Per assicurarti che non ti vengano addebitati servizi indesiderati, puoi eseguire una pulizia eliminando lo stack e le relative risorse.

Eliminare lo stack e delle relative risorse

  1. Dalla CloudFormation console, seleziona lo MyWPTestStack stack.

  2. Scegliere Delete Stack (Elimina stack).

  3. Nel messaggio di conferma visualizzato, scegliere Yes, Delete (Sì, elimina).

Lo stato di MyWPTestStack cambia in DELETE_IN_PROGRESS. Nello stesso modo in cui hai monitorato la creazione dello stack, puoi monitorarne l'eliminazione utilizzando la scheda Event (Evento). Una volta CloudFormation completata l'eliminazione dello stack, lo rimuove dall'elenco.

Complimenti! Hai scelto un modello, hai creato uno stack, hai visualizzato e utilizzato le relative risorse e hai eliminato lo stack e le relative risorse.
