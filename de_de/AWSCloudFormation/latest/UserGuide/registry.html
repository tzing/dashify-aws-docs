Verwenden der AWS CloudFormation Registrierung

Mit der CloudFormation Registrierung können Sie Erweiterungen sowohl öffentliche als auch private verwalten, z. B. Ressourcen, Module und Hooks, die für die Verwendung in Ihrem verfügbar sind AWS-Konto. Derzeit können Sie die folgenden Erweiterungstypen in der - AWS Registrierung verwenden: Ressourcentypen , Module und Hooks . Die Registrierung erleichtert das Erkennen und Bereitstellen von Erweiterungen in Ihren AWS CloudFormation Vorlagen auf die gleiche Weise, wie Sie AWSvon bereitgestellte Ressourcen verwenden.

Öffentliche Erweiterungen und private Erweiterungen

Erweiterungsarten werden als Erweiterungen entweder öffentlich oder privat registriert. Derzeit bietet die Registrierung die folgenden Erweiterungstypen:

  • Ressourcentypen – modellieren und stellen Sie benutzerdefinierte Logik als Ressource bereit, indem Sie Stacks in verwenden CloudFormation.

  • Module – sind eine Möglichkeit für Sie, Ressourcenkonfigurationen für die Aufnahme in Stack-Vorlagen auf transparente, verwaltbare und wiederholbare Weise zu erstellen.

  • Hooks – Überprüfen Sie proaktiv die Konfiguration Ihrer AWS Ressourcen vor der Bereitstellung.

Öffentliche Erweiterungstypen

Öffentliche Erweiterungen sind solche, die öffentlich in der -Registrierung zur Verwendung durch alle CloudFormation Benutzer veröffentlicht werden. Dies schließt Erweiterungen ein, die von AWS und Publishern von Drittanbieter-Erweiterungen veröffentlich wurden.

Es gibt zwei Arten von öffentlichen Erweiterungen:

  • AWS Öffentliche Erweiterungen – Von veröffentlichte Erweiterungen AWS sind immer öffentlich und standardmäßig aktiviert, sodass Sie keine Maßnahmen ergreifen müssen, bevor Sie sie in Ihrem Konto verwenden. Darüber hinaus gilt: AWS steuert die Versionsverwaltung der Erweiterung, sodass Sie immer die neueste verfügbare Version verwenden.

  • Drittanbieter-Erweiterungen – Dies sind Erweiterungen, die für den allgemeinen Gebrauch durch andere Herausgeber als AWSzur Verfügung gestellt werden.

Weitere Informationen finden Sie unter Verwendung von öffentlichen Erweiterungen.

Typen von privaten Erweiterungen

Private Erweiterungen sind solche Erweiterungen von Drittanbietern, die Sie explizit für die Verwendung in Ihrem AWS-Kontofreigeschaltet haben.

Es gibt zwei Arten von Erweiterungen:

  • Aktivierte private Erweiterungen – Sind die lokalen Kopien von Erweiterungen von Drittanbietern, die Sie für Ihr Konto und Ihre Region aktiviert haben. Wenn Sie eine öffentliche Erweiterung eines Drittanbieters aktivieren, CloudFormation erstellt eine lokale Kopie dieser Erweiterung in der Registrierung Ihres Kontos.

  • Registrierte private Erweiterungen – Kann auch private Erweiterungen aktivieren, die nicht in der öffentlichen CloudFormation Registrierung aufgeführt sind. Dies können Erweiterungen sein, die Sie selbst erstellt haben, oder solche, die von Ihrer Organisation oder einem anderen Drittanbieter für Sie freigegeben wurden. Um eine solche private Erweiterung in Ihrem Konto zu verwenden, müssen Sie sie zuerst registrieren. Durch die Registrierung der Erweiterung wird eine Kopie davon in die CloudFormation Registrierung in Ihrem Konto hochgeladen und aktiviert.

Weitere Informationen finden Sie unter Verwendung von privaten Erweiterungen.

Verwalten von Erweiterungen über die CloudFormation Registrierung

Verwenden Sie die - CloudFormation Registrierung, um die Erweiterungen in Ihrem Konto zu verwalten, einschließlich:

  • Anzeigen der verfügbaren und aktivierten Erweiterungen.

  • Private Erweiterungen werden registriert.

  • Aktivieren von Erweiterungen.

So zeigen Sie Erweiterungen in der CloudFormation Konsole an

  1. Melden Sie sich bei der an AWS Management Console und öffnen Sie die - AWS CloudFormation Konsole unter https://console.aws.amazon.com/cloudformation.

  2. Wählen Sie im CloudFormation Navigationsbereich unter Registrierung aus, welche Erweiterungskategorie Sie anzeigen möchten:

    • Öffentliche Erweiterungen zeigt die öffentlichen Erweiterungen an, die in Ihrem Konto verfügbar sind.

      1. Wählen Sie Ihren Erweiterungstyp aus: Ressourcentypen,Module, oder Hooks-Erweiterungen.

      2. Wählen Sie Ihren Publisher: AWS oder Drittanbieter.

      3. Verwenden der Filter Optionen, um die anzuzeigenden Erweiterungen weiter auszuwählen.

    • Aktivierte Erweiterungen zeigt die Erweiterungen an, die in Ihrem Konto aktiviert sind.

      1. Wählen Sie Ihren Erweiterungstyp aus: Ressourcentypen,Module, oder Hooks-Erweiterungen.

      2. Verwenden des Filter-Dropdown-Menü, um die Erweiterungen auszuwählen:

        • AWS – listet Erweiterungen auf, die von veröffentlicht wurden AWS. Erweiterungen veröffentlicht von Erweiterungen AWS sind standardmäßig aktiviert.

        • Drittanbieter – listet alle öffentlichen Erweiterungen von anderen Publishern als denen auf AWS , die Sie in diesem Konto aktiviert haben.

        • Registriet – listet alle privaten Erweiterungen auf, die Sie in diesem Konto aktiviert haben.

    • Publisher zeigt alle öffentlichen Erweiterungen an, die Sie mit diesem Konto veröffentlicht haben. Weitere Informationen finden Sie unter Veröffent von Erweiterungen im Benutzerhandbuch für die Entwicklung von Erweiterungen.

  3. Suchen Sie den Namen der Erweiterung, oder wählen Sie ihn aus, um die Details anzeigen.

Aufzeichnen von Ressourcentypen in AWS Config

Sie können angeben, dass Ihre privaten Ressourcentypen AWS Config automatisch verfolgt und Änderungen an diesen Ressourcen als Konfigurationselemente aufzeichnet. Auf diese Weise können Sie den Konfigurationsverlauf für diese privaten Ressourcentypen anzeigen und zusätzlich AWS-Config-Regeln Regeln schreiben, um bewährte Methoden für die Konfiguration zu überprüfen. AWS Config ist für die Hook-Erweiterung erforderlich.

So lassen Sie Ihre privaten Ressourcentypen AWS Config automatisch verfolgen:

  • Verwalten Sie die Ressourcen über CloudFormation. Dazu gehört das Ausführen aller Vorgänge zum Erstellen, Aktualisieren und Löschen von Ressourcen über CloudFormation.

    Anmerkung

    Wenn Sie eine IAM-Rolle verwenden, um Ihre Stack-Operationen auszuführen, muss diese IAM-Rolle über die Berechtigung zum Aufrufen der folgenden AWS Config Aktionen verfügen:

    • PutResourceConfig

    • DeleteResourceConfig

  • Konfigurieren Sie AWS Config , um alle Ressourcentypen aufzuzeichnen. Weitere Informationen finden Sie im AWS Config -Entwicklerhandbuch unter Aufzeichnen von Konfigurationen für Drittanbieterressourcen.

    Anmerkung

    AWS Config unterstützt nicht die Aufzeichnung privater Ressourcen, die Eigenschaften enthalten, die sowohl als erforderlich als auch schreibgeschützt definiert sind.

    Ressourceneigenschaften, die als schreibgeschützt definiert sind, werden im Schema, das zum Erstellen des AWS Config Konfigurationselements verwendet wird, nicht zurückgegeben. Aus diesem Grund führt das Einschließen einer Eigenschaft, die sowohl als lesegeschützt als auch als erforderlich definiert ist, dazu, dass die Erstellung des Konfigurationselements fehlschlägt, da keine erforderliche Eigenschaft vorhanden sein wird. Um das Schema anzuzeigen, das zum Erstellen des Konfigurationselements verwendet wird, können Sie die -schemaEigenschaft der -DescribeTypeAktion überprüfen.

Weitere Informationen zu Konfigurationselementen finden Sie unter Konfigurationselemente im AWS Config -Entwicklerhandbuch.

Verhindern, dass vertrauliche Eigenschaften in einem Konfigurationselement aufgezeichnet werden

Ihr Ressourcentyp kann Eigenschaften enthalten, die Sie als vertrauliche Informationen betrachten, z. B. Passwörter, Secrets oder andere vertrauliche Daten, die nicht als Teil des Konfigurationselements aufgezeichnet werden sollen. Um zu verhindern, dass eine Eigenschaft im Konfigurationselement aufgezeichnet wird, können Sie diese Eigenschaft in die writeOnlyproperties-Liste in Ihrem Ressourcentypschema aufnehmen. Als writeOnlyproperties aufgelistete Ressourceneigenschaften können vom Benutzer angegeben werden, werden aber nicht von einer read oder list-Anforderung zurückgegeben.

Weitere Informationen finden Sie unter Ressourcenanbieterschema im CloudFormation -Befehlszeilenschnittstellen-Benutzerhandbuch.

Confused-Deputy-Prävention

Das Problem des verwirrten Stellvertreters ist ein Sicherheitsproblem, bei dem eine Entität, die keine Berechtigung zur Durchführung einer Aktion hat, eine privilegiertere Entität zur Durchführung der Aktion zwingen kann. In kann ein AWSserviceübergreifender Identitätswechsel zu einem Confused-Deputy-Problem führen. Ein serviceübergreifender Identitätswechsel kann auftreten, wenn ein Service (der Anruf-Service) einen anderen Service anruft (den aufgerufenen Service). Der Anruf-Service kann so manipuliert werden, dass er seine Berechtigungen verwendet, um auf die Ressourcen eines anderen Kunden zu reagieren, auf die er sonst nicht zugreifen dürfte. Um dies zu verhindern, bietet AWS Tools, mit denen Sie Ihre Daten für alle Services mit Serviceprinzipalen schützen können, die Zugriff auf Ressourcen in Ihrem Konto erhalten haben.

Wir empfehlen die Verwendung der aws:SourceAccount globalen Bedingungskontextschlüssel aws:SourceArn und in Ressourcenrichtlinien, um die Berechtigungen, die einem anderen Service AWS CloudFormation erteilt, auf die Erweiterung zu beschränken. Wenn der aws:SourceArn-Wert nicht die Konto-ID enthält, z. B. den Amazon-Ressourcennamen (ARN) eines Amazon-S3-Buckets, müssen Sie beide globale Bedingungskontext-Schlüssel verwenden, um Berechtigungen einzuschränken. Wenn Sie beide globale Bedingungskontextschlüssel verwenden und der aws:SourceArn-Wert die Konto-ID enthält, müssen der aws:SourceAccount-Wert und das Konto im aws:SourceArn-Wert dieselbe Konto-ID verwenden, wenn sie in der gleichen Richtlinienanweisung verwendet wird. Verwenden Sie aws:SourceArn, wenn Sie nur eine Ressource mit dem betriebsübergreifenden Zugriff verknüpfen möchten. Verwenden Sie aws:SourceAccount, wenn Sie zulassen möchten, dass Ressourcen in diesem Konto mit der betriebsübergreifenden Verwendung verknüpft werden.

Der Wert von aws:SourceArn muss den ARN der Erweiterung verwenden.

Der effektivste Weg, um sich vor dem Confused-Deputy-Problem zu schützen, ist die Verwendung des globalen Bedingungskontext-Schlüssels aws:SourceArn mit dem vollständigen ARN der Ressource. Wenn Sie den vollständigen ARN der Erweiterung nicht kennen oder wenn Sie mehrere Erweiterungen angeben, verwenden Sie den globalen Kontextbedingungsschlüssel aws:SourceArn mit Platzhaltern (*) für die unbekannten Teile des ARN. Beispiel: arn:aws:cloudformation:*:123456789012:*

Anmerkung

Bei Registrierungsservices CloudFormation führt Aufrufe an AWS Security Token Service (AWS STS) durch, um eine Rolle in Ihrem Konto zu übernehmen. Diese Rolle wird für ExecutionRoleArn im RegisterType-Vorgang konfiguriert und für LogRoleArn im LoggingConfig-Vorgang festgelegt.

Das folgende Beispiel zeigt, wie Sie die aws:SourceAccount globalen Bedingungskontextschlüssel aws:SourceArn und in verwenden können AWS CloudFormation , um das Confused-Deputy-Problem zu vermeiden.

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": [
          "resources.cloudformation.amazonaws.com"
        ]
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:SourceAccount": "123456789012"
        },
        "StringLike": {
          "aws:SourceArn": "arn:aws:cloudformation:us-east-1:123456789012:type/resource/Organization-Service-Resource/*"
        }
      }
    }
  ]
}
