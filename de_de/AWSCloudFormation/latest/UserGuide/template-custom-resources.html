Benutzerdefinierte Ressourcen

Anmerkung

Die CloudFormation Registrierung bietet mehrere Vorteile gegenüber benutzerdefinierten Ressourcen, darunter:

  • Unterstützt die Modellierung, Provisionierung und Verwaltung von Anwendungsressourcen von Drittanbietern

  • Unterstützt die Vorgänge Create, Read, Update, Delete und List (CRUDL)

  • Unterstützt die Drifterkennung für Ressourcentypen privater und Drittanbieter

Im Gegensatz zu benutzerdefinierten Ressourcen müssen registrierungsbasierte Ressourcen kein Amazon SNS-Thema oder Lambda-Funktion zur Durchführung von CRUDL-Vorgängen verknüpfen. Weitere Informationen zur CloudFormation Registrierung finden Sie unter Verwenden der AWS CloudFormation Registrierung .

Mit benutzerdefinierten Ressourcen können Sie benutzerdefinierte Bereitstellungslogik in Vorlagen schreiben, die AWS CloudFormation immer dann ausführt, wenn Sie Stacks erstellen, aktualisieren (wenn Sie die benutzerdefinierte Ressource geändert haben) oder löschen. Vielleicht möchten Sie Ressourcen einschließen, die nicht als AWS CloudFormation -Ressourcentypen verfügbar sind. Sie können diese Ressourcen mithilfe von benutzerdefinierten Ressourcen einschließen. Auf diese Weise können Sie alle zugehörigen Ressourcen weiterhin in einem einzigen Stack verwalten.

Verwenden Sie den Ressourcentyp AWS::CloudFormation::CustomResource oder Custom::MyCustomResourceTypeName, um benutzerdefinierte Ressourcen in Ihren Vorlagen zu definieren. Für benutzerdefinierte Ressourcen ist eine Eigenschaft erforderlich: das Service-Token, das angibt, wohin AWS CloudFormation Anforderungen sendet, z. B. an ein Amazon-SNS-Thema.

Anmerkung

Wenn Sie die Funktion VPC-Endpunkt verwenden, müssen benutzerdefinierte Ressourcen in der VPC Zugriff auf AWS CloudFormation-spezifische S3-Buckets haben. Benutzerdefinierte Ressourcen müssen Antworten an eine vorsignierte Amazon S3-URL senden. Wenn sie keine Antworten an Amazon S3 senden können, empfängt AWS CloudFormation keine Antwort und der Stack-Vorgang schlägt fehl. Weitere Informationen finden Sie unter Einrichten von VPC-Endpunkten für AWS CloudFormation.

So funktionieren benutzerdefinierte Ressourcen

Alle Aktionen, die für eine benutzerdefinierte Ressource ausgeführt werden, bestehen aus drei Teilen.

template developer

Erstellt eine Vorlage, die einen benutzerdefinierten Ressourcentyp umfasst. Der template developer gibt das Service-Token sowie alle anderen Eingabedaten in der Vorlage an.

custom resource provider

Besitzt die benutzerdefinierte Ressource und bestimmt, wie Anforderungen von AWS CloudFormation verarbeitet und darauf geantwortet werden soll. Der custom resource provider muss ein Service-Token bereitstellen, das der template developer verwendet.

AWS CloudFormation

Während eines Stack-Vorgangs wird eine Anforderung an das Service-Token gesendet, das in der Vorlage angegeben ist. Dann wird auf eine Antwort gewartet, bevor der Stack-Vorgang fortgesetzt wird.

Der template developer und custom resource provider können dieselbe Person oder Entität sein, der Prozess ist in jedem Fall derselbe. In den folgenden Schritten wird der allgemeine Prozess beschrieben:

  1. Der template developer definiert eine benutzerdefinierte Ressource in seiner Vorlage, die ein Service-Token und alle Eingabeparameter enthält. In Abhängigkeit von der benutzerdefinierten Ressource sind die Eingabedaten möglicherweise erforderlich; das Service-Token ist jedoch immer erforderlich.

    Das Service-Token gibt an, wohin AWS CloudFormation Anforderungen sendet, z. B. ein Amazon-SNS-Thema-ARN oder ein AWS Lambda-Funktions-ARN. Weitere Informationen finden Sie unter AWS::CloudFormation::CustomResource. Das Service-Token und die Struktur der Eingabedaten werden vom custom resource provider definiert.

  2. Immer dann, wenn eine Person die Vorlage zum Erstellen, Aktualisieren oder Löschen einer benutzerdefinierten Ressource verwendet, sendet AWS CloudFormation eine Anforderung an das angegebene Service-Token. Das Service-Token muss in der Region sein, in der Sie den Stack erstellen.

    AWS CloudFormation schließt in die Anforderungen Informationen ein, z. B. den Anforderungstyp und eine vorsignierte URL von Amazon Simple Storage Service, an die die benutzerdefinierte Ressource Antworten sendet. Weitere Informationen zu den in der Anforderung enthaltenen Informationen finden Sie unter Anforderungsobjekte für benutzerdefinierte Ressourcen.

    Die folgenden Beispieldaten zeigen, welche Informationen AWS CloudFormation in eine Anforderung einschließt:

    {
   "RequestType" : "Create",
   "ResponseURL" : "http://pre-signed-S3-url-for-response",
   "StackId" : "arn:aws:cloudformation:us-west-2:123456789012:stack/stack-name/guid",
   "RequestId" : "unique id for this create request",
   "ResourceType" : "Custom::TestResource",
   "LogicalResourceId" : "MyTestResource",
   "ResourceProperties" : {
      "Name" : "Value",
      "List" : [ "1", "2", "3" ]
   }
}
    Anmerkung

    In diesem Beispiel erlaubt ResourceProperties, dass AWS CloudFormation eine benutzerdefinierte Nutzlast erstellt, die an die Lambda-Funktion gesendet wird.

  3. Der benutzerdefinierte Ressourcenanbieter verarbeitet die AWS CloudFormation-Anforderung und gibt die Antwort SUCCESS oder FAILED an die vorsignierte URL zurück. Der custom resource provider gibt die Antwort in einer JSON-formatierten Datei zurück und lädt diese zu der vorsignierten S3-URL hoch. Weitere Informationen finden Sie unter Hochladen von Objekten mithilfe von vorsignierten URLs im Benutzerhandbuch für Amazon Simple Storage Service.

    Der custom resource provider kann in der Antwort auch Name-Wert-Paare angeben, auf die der template developer zugreifen kann. Die Antwort kann beispielsweise Ausgabedaten enthalten, wenn die Anforderung erfolgreich war, oder eine Fehlermeldung, wenn die Anforderung fehlgeschlagen ist. Weitere Informationen zu Antworten finden Sie unter Benutzerdefinierte Ressourcenantwortobjekte.

    Wichtig

    Wenn die Namen-Wert-Paare vertrauliche Informationen enthalten, sollten Sie das NoEcho-Feld benutzen, um die Ausgabe der benutzerdefinierten Ressource zu maskieren. Andernfalls sind die Werte durch APIs sichtbar, die Eigenschaftswerte (wie DescribeStackEvents) enthalten.

    Weitere Informationen zum Maskieren vertraulicher Daten mit NoEcho enthält die bewährte Methode Keine Anmeldeinformationen in Ihre Vorlagen einbetten.

    Der custom resource provider ist für das Erkennen und Beantworten der Anforderung verantwortlich. Für Amazon-SNS-Benachrichtigungen beispielsweise muss der benutzerdefinierte Ressourcenanbieter auf Benachrichtigungen hören und reagieren, die zu einem bestimmten Themen-ARN gesendet werden. AWS CloudFormation wartet und hört auf eine Antwort am vordefinierten URL-Standort.

    In den folgenden Beispieldaten wird gezeigt, was eine benutzerdefinierte Ressourcen in eine Antwort einschließen kann:

    {
   "Status" : "SUCCESS",
   "PhysicalResourceId" : "TestResource1",
   "StackId" : "arn:aws:cloudformation:us-west-2:123456789012:stack/stack-name/guid",
   "RequestId" : "unique id for this create request",
   "LogicalResourceId" : "MyTestResource",
   "Data" : {
      "OutputName1" : "Value1",
      "OutputName2" : "Value2",
   }
}

  4. Nach Erhalt einer SUCCESS-Antwort setzt AWS CloudFormation den Stack-Vorgang fort. Wenn keine FAILED-Antwort oder gar keine Antwort zurückgegeben wird, schlägt der Vorgang fehl. Alle Ausgabedaten von der benutzerdefinierten Ressource werden am Speicherort der vorsignierten URL gespeichert. Der template developer kann die Daten mithilfe der Fn::GetAtt-Funktion abrufen.
