Durchführung eines Machbarkeitsnachweises für Amazon Redshift

Amazon Redshift ist ein schneller, skalierbarer Data-Warehouse-Service, mit dem Sie im Zusammenspiel mit Ihren vorhandenen BI-Tools (Business Intelligence) alle Ihre Daten einfach und wirtschaftlich analysieren können. Amazon Redshift bietet schnelle Leistung in einem kostengünstigen Cloud-Data-Warehouse. Die Lösung nutzt fortschrittliche Abfrageoptimierung, schnelle Zwischenspeicherung, Speicherung in Spaltenanordnung auf lokalen Hochleistungslaufwerken und massiv parallele Abfrageausführung.

In den folgenden Abschnitten wird ein Framework zur Erstellung eines Machbarkeitsnachweises mit Amazon Redshift vorgestellt. Das Framework gibt Ihnen bewährte Methoden zum Entwerfen und Betreiben eines sicheren und kostengünstigen Data Warehouse mit hoher Leistung an die Hand. Dieser Leitfaden basiert auf der Prüfung der Designs von Tausenden Kundenarchitekturen in unterschiedlichsten Geschäftssegmenten und Einsatzszenarien. Wir haben die Kundenerfahrungen ausgewertet, um die hier vorgestellten bewährten Methoden zu entwickeln, mit deren Hilfe Sie Kriterien zum Evaluieren der Workloads eines Data Warehouses entwickeln können.

Übersicht über den Prozess

Die Durchführung eines Machbarkeitsnachweises ist ein dreistufiger Prozess:

  1. Identifizieren Sie die Ziele des Machbarkeitsnachweises – Sie können rückwärts von Ihren geschäftlichen Anforderungen und Erfolgskriterien arbeiten und diese in einen technischen Machbarkeitsnachweis-Projektplan umsetzen.

  2. Richten Sie die Umgebung für den Machbarkeitsnachweis ein – die Einrichtung besteht im Wesentlichen aus wenigen Klicks auf einige Schaltflächen zur Erstellung Ihrer Ressourcen. Die Erstellung einer Data-Warehouse-Umgebung mit geladenen Daten dauert nur wenige Minuten.

  3. Füllen Sie den Machbarkeitsnachweis-Projektplan aus, um sicherzustellen, dass die Ziele erreicht werden.

In den folgenden Abschnitten werden die Details jedes einzelnen Schritts erläutert.

Ermitteln der Geschäftsziele und Erfolgskriterien

Das Identifizieren der Ziele des Machbarkeitsnachweises hat eine wichtige Funktion, wenn Sie die im Rahmen des Evaluierungsprozesses zu messenden Kennzahlen festlegen. Die Evaluierungskriterien sollten die aktuellen Skalierungsherausforderungen, Erweiterungen zur Verbesserung des Data-Warehouse-Kundenerlebnisses und Methoden zum Umgang mit den aktuellen betrieblichen Problembereichen umfassen. Die Ziele des Machbarkeitsnachweises können Sie mithilfe der folgenden Fragen ermitteln:

  • Was sind die Ziele der Skalierung des -Data-Warehouses?

  • Welche Service Level Agreements gibt es, deren Bestimmungen Sie verbessern möchten?

  • Welche neuen Datensätze möchten Sie in das Data Warehouse aufnehmen?

  • Für welche geschäftskritischen SQL-Abfragen müssen Tests und Messungen vorgenommen werden? Achten Sie darauf, alle Aspekte der SQL-Komplexität wie unterschiedliche Abfragetypen (z. B. Select, Insert, Update und Delete) zu berücksichtigen.

  • Welche grundlegenden Workload-Typen möchten Sie testen? Beispiele sind ETL-Workloads (Extrakt, Transform, Load), Berichtsabfragen und Batch-Extraktionen.

Nachdem Sie diese Fragen beantwortet haben, sollten Sie ein SMART-Ziele und Erfolgskriterien zum Erstellen des Machbarkeitsnachweises formulieren können. Informationen zum Festlegen von Zielen finden Sie unter SMART criteria in Wikipedia.

Einrichtung des Machbarkeitsnachweises

Da Hardwarebereitstellung, Networking und Softwareinstallation im lokalen Data Warehouse nicht mehr erforderlich sind, war es nie einfacher Amazon Redshift mit Ihrem eigenen Datensatz zu testen. Viele der Entscheidungen zur Größenanpassung und Schätzungen, die bisher erforderlich, können mittlerweile mit nur einem Klick erledigt werden. Sie können die Größe Ihres Clusters flexibel ändern oder das Verhältnis zwischen Speicher und Rechenleistung anpassen.

Im Wesentlichen erfolgt die Einrichtung der Machbarkeitsnachweis-Umgebung von Amazon Redshift in zwei Schritten: Den Start eines Data Warehouse und die anschließende Konvertierung des Schemas und der Datensätze für die Evaluierung.

Auswahl einer anfänglichen Clustergröße

Sie können den Knotentyp und die Anzahl der Knoten über die Amazon Redshift Konsole auswählen. Es wird empfohlen, als Teil Ihres Machbarkeitsnachweisplans auch die Größenanpassung des Clusters zu testen. Die anfängliche Größenanpassung Ihres Clusters wird wie folgt durchgeführt:

  1. Melden Sie sich bei der AWS Management Console an und öffnen Sie die Amazon-Redshift-Konsole unter https://console.aws.amazon.com/redshift/.

  2. Wählen Sie im Navigationsmenü Create cluster (Cluster erstellen) aus, um die Konfigurationsseite zu öffnen.

  3. Geben Sie unter Cluster identifier (Cluster-ID) den Namen des Clusters ein.

  4. Im folgenden Schritt wird eine Amazon-Redshift-Konsole beschrieben, die in einer AWS-Region ausgeführt wird, die RA3-Knotentypen unterstützt. Eine Liste von AWS-Regionen, die RA3-Knotentypen unterstützen, finden Sie unter Übersicht über RA3-Knotentypen im Amazon-Redshift-Verwaltungshandbuch.

    Wenn Sie nicht wissen, wie groß Ihr Cluster sein sollte, wählen Sie Help me choose (Hilfe bei der Auswahl) aus. Dadurch wird ein Größenbestimmungsrechner gestartet, der Ihnen Fragen zur Größe und zu Abfragemerkmalen der Daten stellt, die Sie in Ihrem Data Warehouse speichern möchten. Wenn Sie die erforderliche Größe Ihres Clusters kennen (d. h. den Knotentyp und die Anzahl der Knoten), wählen Sie I'll choose (Ich entscheide) aus. Wählen Sie den Node type (Knotentyp) und die Anzahl der Nodes (Knoten) aus, um die Größe Ihres Clusters für den Machbarkeitsnachweis zu bestimmen.

    Anmerkung

    Wenn Ihre Organisation berechtigt ist und Ihr Cluster in einer AWS-Region erstellt wird, in der Amazon Redshift Serverless nicht verfügbar ist, können Sie möglicherweise im Rahmen des kostenlosen Testprogramms von Amazon Redshift einen Cluster erstellen. Wählen Sie entweder Produktion oder Kostenlose Testversion als Antwort auf die Frage: Wofür möchten Sie diesen Cluster verwenden? Wenn Sie Kostenlose Testversion auswählen, erstellen Sie eine Konfiguration mit dem Knotentyp dc2.large. Weitere Informationen zur Auswahl einer kostenlosen Testversion finden Sie unter Kostenloses Testprogramm für Amazon Redshift. Eine Liste der AWS-Regionen, in denen Amazon Redshift Serverless verfügbar ist, finden Sie in der Übersicht über die Endpunkte für die Redshift-Serverless-API in der Allgemeine Amazon Web Services-Referenz..

  5. Nachdem Sie alle erforderlichen Clustereigenschaften eingegeben haben, wählen SieCreate cluster (Cluster erstellen) aus, um Ihr Data Warehouse zu starten.

    Weitere Informationen zum Erstellen von Clustern mit der Amazon-Redshift-Konsole finden Sie unter Erstellen eines Clusters im Amazon-Redshift-Verwaltungshandbuch.

Konvertieren des Schemas und Einrichten der Datensätze für den Machbarkeitsnachweis

Wenn Sie kein Data Warehouse haben, überspringen Sie diesen Abschnitt und gehen Sie zum Handbuch Erste Schritte mit Amazon Redshift. Das Handbuch Erste Schritte mit Amazon Redshift bietet ein Tutorial zum Erstellen eines Clusters und Beispiele für das Einrichten von Daten in Amazon Redshift.

Bei der Migration eines vorhandenen Data Warehouse können Sie das Schema, den Code und die Daten mit dem AWS Schema Conversion Tool und dem AWS Database Migration Service konvertieren. Die Auswahl der Tools hängt von der Quelle Ihrer Daten und optionalen laufenden Replikationen ab. Weitere Informationen finden Sie unter Was ist das AWS Schema Conversion Tool? im AWS-Schema-Conversion-Tool-Benutzerhandbuch und unter Was ist AWS Database Migration Service? im AWS Database Migration Service-Benutzerhandbuch. Die folgenden Informationen erleichtern das Einrichten der Daten in Amazon Redshift::

  • Migrate Your Data Warehouse to Amazon Redshift Using the AWS Schema Conversion Tool – Dieser Blogbeitrag stellt eine Übersicht darüber bereit, wie Sie die AWS SCT-Datenextraktoren verwenden, um Ihr vorhandenes Data Warehouse nach Amazon Redshift zu migrieren. Das AWS SCT-Tool kann Ihre Daten von vielen Legacy-Plattformen (wie Oracle, Greenplum, Netezza, Teradata, Microsoft SQL Server oder Vertica) migrieren.

  • Optional können Sie auch den AWS Database Migration Service für laufende Replikationen von geänderten Daten von der Quelle verwenden. Weitere Informationen finden Sie unter Verwendung einer Amazon-Redshift-Datenbank als Ziel für den AWS Database Migration Service im AWS Database Migration Service-Benutzerhandbuch.

Amazon Redshift ist ein Verwaltungssystem für relationale Datenbanken (Relational Database Management System, RDBMS). Es kann viele Arten von Datenmodellen ausführen, zum Beispiel Sternschemata, Schneeflockenschemata, Data-Vault-Modelle und einfache, flache oder normalisierte Tabellen. Nachdem Sie Ihre Schemata in Amazon Redshift eingerichtet haben, können Sie sofort die Vorteile der massiv parallelen Verarbeitung und der Spaltendatenspeicherung für schnelle analytische Abfragen nutzen. Weitere Informationen zu den Schematypen finden Sie in den Wikipedia-Artikeln Star schema, Snowflake schema und Data vault modeling.

Checkliste für eine vollständige Evaluierung

Stellen Sie sicher, dass eine vollständige Evaluierung alle Ihre Data-Warehouse-Anforderungen erfüllt. Berücksichtigen Sie die folgenden Elemente in Ihre Erfolgskriterien:

  • Datenladezeit – Die Verwendung des Befehls COPY ist eine häufige Methode, um die Dauer eines Ladevorgangs zu testen. Weitere Informationen finden Sie unter Bewährte Methoden für Amazon Redshift zum Laden von Daten.

  • Durchsatz des Clusters – Das Messen der Abfragen pro Stunde ist eine häufig angewendete Methode zum Ermitteln des Durchsatzes. Richten Sie zu diesem Zweck einen Test ein, der typische Abfragen für die Workload ausführt.

  • Datensicherheit – Sie können mit Amazon Redshift Daten im Ruhezustand und während der Übertragung auf einfache Weise verschlüsseln. Zudem gibt es eine Reihe von Optionen zum Verwalten von Schlüsseln. Amazon Redshift unterstützt außerdem die Integration von Single Sign-On. Die Preise für Amazon Redshift umfassen integrierte Sicherheit, Datenkomprimierung, Sicherungsspeicher und Datenübertragung.

  • Integration von Drittanbieter-Tools – Sie können für die Integration mit Business-Intelligence-Tools und anderen externen Tools eine JDBC- oder ODBC-Verbindung verwenden.

  • Interoperabilität mit anderen AWS-Services – Amazon Redshift ermöglicht die Integration mit anderen AWS-Services, wie Amazon EMR, Amazon QuickSight, AWS Glue, Amazon S3 und Amazon Kinesis. Sie können diese Integration beim Konfigurieren und Verwalten des Data Warehouse verwenden.

  • Backups und Snapshots – Backups und Snapshots werden automatisch erstellt. Sie können auch jederzeit oder nach Zeitplan einen zeitbezogenen Snapshot erstellen. Sie sollten für Ihre Evaluierung einen Snapshot verwenden und einen zweiten Cluster erstellen. Evaluieren Sie, ob Ihre Abteilungen für Entwicklung und Test den Cluster verwenden können.

  • Größenanpassung – Ihre Evaluierung sollte eine Erhöhung der Zahl oder der Typen von Amazon-Redshift-Knoten umfassen. Prüfen Sie, ob der Workload-Durchsatz vor und nach einer Größenanpassung jede Variabilität des Volumens Ihrer Workload erfüllt. Weitere Informationen finden Sie unter Größenanpassung von Clustern in Amazon Redshift im Amazon-Redshift-Verwaltungshandbuch.

  • Nebenläufigkeitsskalierung – Mit dieser Funktion können Sie auf die Variabilität des Datenverkehrsvolumens in Ihrem Data Warehouse reagieren. Mit der Nebenläufigkeitsskalierung können Sie praktisch unbegrenzt gleichzeitige Benutzer und gleichzeitige Abfragen unterstützen, bei gleichbleibend schneller Abfrageleistung. Weitere Informationen finden Sie unter Arbeiten mit Nebenläufigkeitsskalierung.

  • Automatisches Workload-Management (WLM) – Priorisieren Sie dank dem automatischen WLM Ihre geschäftskritischen Abfragen gegenüber anderen Abfragen. Versuchen Sie, Warteschlangen basierend auf Ihren Workloads einzurichten (z. B. eine Warteschlange für ETL und eine Warteschlange für die Berichterstellung). Aktivieren Sie anschließend das automatische WLM, um die Nebenläufigkeit und Speicherressourcen dynamisch zuzuweisen. Weitere Informationen finden Sie unter Implementieren von automatischem WLM.

  • Amazon Redshift Advisor – Advisor entwickelt maßgeschneiderte Empfehlungen zur Steigerung der Leistung und Kostenoptimierung durch Analyse des Workloads und Nutzungsmetriken für Ihr Cluster. Melden Sie sich bei der Amazon-Redshift-Konsole an, um Advisor-Empfehlungen anzuzeigen. Weitere Informationen finden Sie unter Arbeiten mit Empfehlungen von Amazon Redshift Advisor.

  • Tabellendesign – Amazon Redshift bietet hervorragende Leistung für die meisten Workloads. Wenn Sie eine Tabelle erstellen, lautet der Standard-Sortierschlüssel und -Verteilungsschlüssel AUTO. Weitere Informationen finden Sie unter Arbeiten mit automatischer Tabellenoptimierung.

  • Support – Wir empfehlen ausdrücklich, im Rahmen Ihrer Evaluierung den AWS Support zu evaluieren. Sie sollten außerdem mit Ihrem Account Manager über Ihren Machbarkeitsnachweis sprechen. AWS kann Ihnen technische Hilfestellungen und Gutschriften für den Machbarkeitsnachweis bereitstellen, wenn Sie dazu berechtigt sind. Wenn Sie danach noch offene Fragen haben, können Sie direkt mit dem Amazon-Redshift-Team sprechen. Übermitteln Sie hierfür ein Formular für eine Support-Anfrage für Ihren Amazon-Redshift-Machbarkeitsnachweis.

  • Lake-House-Integration – Testen die Sie diese Amazon-Redshift-Spectrum-Funktion, die nativ integriert und sofort einsatzbereit ist. Mit Redshift Spectrum können Sie Ihr Data Warehouse in Ihren Data Lake erweitern und Abfragen für Petabyte an Daten in Amazon S3 mit Ihrem vorhandenen Cluster ausführen. Weitere Informationen finden Sie unter Abfrage externer Daten mit Amazon Redshift Spectrum.

Entwicklung eines Projektplans für Ihre Evaluation

Einige der folgenden Techniken zum Erstellen von Abfrage-Benchmarks sind möglicherweise auch für Ihre Amazon-Redshift-Evaluierung sinnvoll:

  • Stellen Sie eine Liste von Abfragen für jede Laufzeitkategorie zusammen. Durch eine ausreichende Anzahl (z. B. 30 pro Kategorie) können Sie sicherstellen, dass die Evaluierung der tatsächlichen Data-Warehouse-Implementierung entspricht. Fügen Sie eine eindeutige Kennung hinzu, um jede in die Evaluierung aufgenommene Abfrage einer der Kategorien zuzuordnen, die Sie für die Evaluierung festgelegt haben. Sie können diese eindeutigen Kennungen dann verwenden, um den Durchsatz aus den Systemtabellen zu ermitteln.

    Sie können auch eine Abfragegruppe erstellen, um die Evaluierungsabfragen zu ordnen. Wenn Sie beispielsweise eine Kategorie namens "Reporting" für die Evaluierung eingerichtet haben, können Sie ein Codierungssystem bereitstellen und die Evaluierungsabfragen mit dem Wort "Report" kennzeichnen. Dann können Sie die individuellen Abfragen innerhalb der Kategorie "Reporting" als R1, R2 usw. kennzeichnen. Das folgende Beispiel illustriert diese Herangehensweise.

    
SELECT 'Reporting' AS query_category, 'R1' as query_id, * FROM customers;              

    
SELECT query, datediff(seconds, starttime, endtime)
  FROM stl_query 
  WHERE 
  querytxt LIKE '%Reporting%' 
  and starttime >= '2018-04-15 00:00' 
  and endtime < '2018-04-15 23:59';

    Sobald Sie eine Abfrage einer Evaluierungskategorie zugeordnet haben, können Sie eine eindeutige Kennung verwenden, um den Durchsatz der Systemtabellen für jede Kategorie zu bestimmen.

  • Testen Sie den Durchsatz mit Benutzer- oder ETL-Abfragen aus der Vergangenheit, die im vorhandenen Data Warehouse unterschiedliche Laufzeiten haben. Sie können ein Dienstprogramm für Lasttests verwenden, z. B. das Open-Source-Programm JMeter oder ein benutzerdefiniertes Dienstprogramm. Falls Sie ein solches Programm verwenden, sollte es folgende Funktionen bieten:

    • Es berücksichtigt die Netzwerkübertragungszeit.

    • Es evaluiert die Ausführungszeit basierend auf dem Durchsatz der internen Systemtabellen. Weitere Informationen über die entsprechende Vorgehensweise finden Sie unter Analysieren des Abfragezusammenfassung.

  • Identifizieren Sie alle Permutationen, die Sie im Rahmen der Evaluierung testen wollen. Die folgende Liste stellt einige gebräuchliche Variablen bereit:

    • Cluster-Größe

    • Knotentyp

    • Lastprüfungsdauer

    • Einstellungen für Parallelbetrieb

  • Reduzieren Sie die Kosten für Ihren Machbarkeitsnachweis, indem Sie Ihren Cluster außerhalb der Öffnungszeiten und am Wochenende anhalten. Wenn ein Cluster angehalten wird, wird auch die On-Demand-Computing-Fakturierung unterbrochen. Um Tests auf dem Cluster auszuführen, setzen Sie die sekundengenauen Abrechnung fort. Sie können auch einen Zeitplan erstellen, um den Cluster automatisch anzuhalten und fortzusetzen. Weitere Informationen finden Sie unter Anhalten und Fortsetzen von Clustern im Amazon-Redshift-Verwaltungshandbuch.

Sie sind nun bereit, Ihren Projektplan auszuführen und die Ergebnisse zu evaluieren.
