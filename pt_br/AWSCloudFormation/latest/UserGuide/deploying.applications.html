Implantar aplicações no Amazon EC2 com o AWS CloudFormation

Você pode usar o AWS CloudFormation para instalar, configurar e iniciar aplicações automaticamente em instâncias do Amazon EC2. Isso permite que você duplique implantações e atualize instalações existentes facilmente sem se conectar diretamente à instância, o que pode economizar muito tempo e esforço.

O CloudFormation inclui um conjunto de scripts auxiliares (cfn-init, cfn-signal, cfn-get-metadata e cfn-hup) que são baseados em cloud-init. Você chama esses scripts auxiliares de seus modelos do CloudFormation para instalar, configurar e atualizar os aplicativos em instâncias do Amazon EC2 que estão no mesmo modelo.

A descrição a seguir descreve como criar um modelo que ativa uma pilha LAMP usando scripts cfn auxiliares para instalar, configurar e iniciar o Apache, o MySQL e o PHP. Você começará com um simples modelo que define uma instância básica do Amazon EC2 executando o Amazon Linux e, em seguida, continuará adicionando ao modelo até que ele descreva uma pilha LAMP completa.

Finalmente, a seção de saída informa o URL público do servidor Web.

Você se baseará no modelo de Amazon EC2 básico anterior para instalar automaticamente o Apache, o MySQL e o PHP. Para instalar os aplicativos, você adicionará uma propriedade UserData e uma propriedade Metadata. No entanto, o modelo não irá configurar e iniciar os aplicativos até a próxima seção.

Configuração de LAMP

Agora que temos um modelo que instala Linux, Apache, MySQL e PHP, precisaremos expandir o modelo para que ele configure e execute o Apache, o MySQL e o PHP automaticamente. No exemplo a seguir, expandimos na seção Parameters, o recurso AWS::CloudFormation::Init e a propriedade UserData para concluir a configuração. Como no modelo anterior, as seções marcadas com reticências (...) são omitidas para brevidade. Adições ao modelo são mostradas em texto vermelho em itálico.

O exemplo define os parâmetros DBUsername e DBPassword com a propriedade NoEcho definida como true. Se você definir o atributo NoEcho como true, o CloudFormation retornará o valor do parâmetro mascarado como asteriscos (*****) para qualquer chamada que descreva a pilha ou os eventos de pilha, exceto informações armazenadas nos locais especificados abaixo.

Importante

O uso do atributo NoEcho não mascara informações armazenadas no seguinte:

  • A seção de modelo de Metadata. O CloudFormation não transforma, modifica nem edita nenhuma informação incluída na seção Metadata. Para ter mais informações, consulte Metadados.

  • A seção de modelo de Outputs. Para ter mais informações, consulte Outputs.

  • O atributo Metadata de uma definição de recurso. Para ter mais informações, consulte Atributo Metadata.

É altamente recomendável não usar esses mecanismos para incluir informações confidenciais, como senhas ou segredos.

Importante

Em vez de incorporar informações confidenciais diretamente em modelos do CloudFormation, recomendamos usar os parâmetros dinâmicos no modelo da pilha para fazer referência a informações confidenciais que são armazenadas e gerenciadas de forma externa ao CloudFormation, como no AWS Systems Manager Parameter Store ou no AWS Secrets Manager.

Para obter mais informações, consulte as práticas recomendadas do Não incorporar credenciais em seus modelos.

Por exemplo, você pode alterar o aplicativo PHP de exemplo e executar uma atualização da pilha para implantar a alteração.

Para executar os comandos MySQL após a conclusão da instalação, o exemplo adiciona outro conjunto de configurações para executar os comandos. Os conjuntos de configurações são úteis quando você tem uma série de tarefas que devem ser concluídas em uma ordem específica. O exemplo primeiro executa o conjunto de configurações Install e, em seguida, o conjunto de configurações Configure. O conjunto de configurações Configure especifica a senha raiz do banco de dados e cria um banco de dados. Na seção de comandos, os comandos são processados em ordem alfabética por nome, de modo que o exemplo adiciona um número antes de cada nome de comando para indicar a ordem de execução desejada.

Atributo CreationPolicy

Por fim, você precisa de uma forma para instruir o CloudFormation a concluir a criação da pilha somente depois que todos os serviços (como o Apache e o MySQL) estiverem em execução, e não depois que todos os recursos da pilha forem criados. Em outras palavras, se você usar o modelo na seção anterior para ativar uma pilha, o CloudFormation definirá o status dela como CREATE_COMPLETE depois de criar todos os recursos com êxito. No entanto, se houver falha na inicialização de um ou mais serviços, o CloudFormation ainda definirá o status da pilha como CREATE_COMPLETE. Para evitar a alteração do status para CREATE_COMPLETE até que todos os serviços tenham sido iniciados com êxito, você pode adicionar um atributo Atributo CreationPolicy à instância. Esse atributo coloca a instância no status CREATE_IN_PROGRESS até que o CloudFormation receba o número necessário de sinais de êxito ou o tempo limite seja excedido, portanto, você pode controlar quando a instância foi criada com êxito.

O exemplo a seguir adiciona uma política de criação à instância Amazon EC2 para garantir que cfn-init conclua a instalação e a configuração do LAMP antes que a criação da pilha seja concluída. Em conjunto com a política de criação, o exemplo precisa executar o script auxiliar cfn-signal para sinalizar para o CloudFormation quando todos os aplicativos estiverem instalados e configurados.
